## 设计原则声明

本系统的评分逻辑基于以下核心理论（来源：康哲BP体系文档）：

> **评价原则一**：Outcome（结果）与 Contribution（贡献）必须分离评价。
> - 组织BP：结果导向，评价"目标是否兑现"
> - 个人BP：贡献导向，评价"这一段责任链是否跑通"

> **评价原则二**：激励不是评价的自动结果，而是基于评价的管理决策。
> - 本系统只输出"贡献度评分"，不直接输出奖金系数
> - 最终激励决策由管理者结合评分结果做出，不由系统自动计算

本系统聚焦于"贡献度评分"这一环节，不涉及最终激励决策。

## 一、范围定义

### 1.1 做什么（IN SCOPE）

- 接受任意组织名称（任意层级）作为输入，定位该组织
- 拉取该组织的完整BP树（组织层 → 部门层 → 个人层）
- 对同级BP进行相对权重评分（总分100分，反映各BP的相对重要性）
- 对承接关系逐层分解分值池，直到个人层
- 输出每个承接人的最终得分及评分理由
- 支持人工确认和微调，并联动更新下级分值

### 1.2 不做什么（OUT OF SCOPE）

- ❌ 不修改BP系统中的任何数据（只读）
- ❌ 不自动发送评分结果给任何人
- ❌ 不评判BP的"绝对好坏"——只做同级相对价值排序
- ❌ 不处理历史周期（只处理指定周期）
- ❌ 不直接输出奖金比例或系数（激励决策由管理者做）
- ❌ 不评价个人BP的"达成率"（本系统评价的是贡献度，不是完成率）

## 三、评分算法规范（核心）

### 3.1 总体流程

```
用户输入：组织名 + BP周期
    ↓
Step 1：searchByName → 定位组织节点（groupId）
    ↓
Step 2：getAllPeriod → 确认周期ID
    ↓
Step 3：getSimpleTree → 获取该组织的一级BP列表
    ↓
Step 4：getGoalAndKeyResult → 获取每个BP的目标/KR/举措详情
    ↓
Step 5：AI同级评分 → 为N个同级BP分配100分权重
    ↓
Step 6：taskDepts(role=undertaker) → 识别每个BP的承接方
    ↓
Step 7：递归进入承接方 → 重复Step 3-6
    ↓
Step 8：终止于个人层（无下级承接）
    ↓
Step 9：自底向上汇总每人得分
    ↓
Step 10：生成评分报告 + 标注低置信度项
    ↓
Step 11：人工确认/微调 → 输出最终版本
```

### 3.2 第一步：同级BP权重评分

**目的**：给同一组织下的N个BP分配相对权重，总和 = 100分。

**输入**：同一组织的所有BP详情（目标文本、KR列表、举措列表、上级BP信息）

#### 评分维度表

| 维度 | 权重（默认） | AI判断依据 |
|---|---|---|
| **战略对齐度** | 40% | 该BP目标与上级BP/组织战略的关键词重叠度、因果逻辑链紧密程度；是否直接承接了上级的核心举措意图 |
| **成果可衡量性** | 20% | KR是否有明确数字锚点、时间节点、可验收标准；"提升"类描述无具体数字的扣分 |
| **举措覆盖度** | 20% | 举措数量与质量能否逻辑上支撑目标达成；举措与目标之间是否有明确因果链 |
| **影响面与辐射范围** | 20% | 承接部门数量、跨中心/跨部门协同程度、影响人员规模估算 |

**约束**：
- 权重之和必须 = 100%（由yaml配置保证，参见第七节）
- N个BP的分值之和必须 = 100（允许误差 < 0.1分，由代码四舍五入保证）
- 单个BP最低分 ≥ 1分（避免完全忽视某BP）

### 3.3 第二步：分值池逐层分配

**目的**：将每个BP的分值，按承接关系向下分解到个人。

**核心规则**：
- 某BP得分X分 → X分是其所有承接方共享的分值池
- 承接方的分值 = X × 该承接方的贡献度比例
- 所有承接方的贡献度比例之和 = 100%
- 递归执行，直到个人层

#### 评分维度表

| 维度 | 权重（默认） | AI判断依据 |
|---|---|---|
| **目标承接准确性** | 45% | 下级BP目标是否真实对应上级BP的举措意图，有无偏移或替换；是否存在"换皮不换骨"的虚假承接 |
| **成果对上级贡献** | 35% | 下级BP的成果能否直接推动上级BP目标实现；成果与上级KR之间是否有明确因果链 |
| **举措完整性** | 20% | 下级BP的举措是否完整覆盖了所承接的上级举措范围；有无明显遗漏 |

#### 递归规则

```
function distribute(bp_node, bp_score):
    undertakers = get_undertakers(bp_node)  # 获取所有承接方
    
    if len(undertakers) == 0:
        # 无承接方：分值保留在当前层，标注[无承接]
        return mark_no_undertaker(bp_node, bp_score)
    
    if len(undertakers) == 1:
        # 单一承接方：100%分配，无需AI比较
        undertaker_score = bp_score
        recurse(undertakers[0], undertaker_score)
        return
    
    # 多个承接方：AI评估贡献度比例
    ratios = ai_score_undertakers(bp_node, undertakers)  # 比例之和=1.0
    for undertaker, ratio in zip(undertakers, ratios):
        undertaker_score = bp_score * ratio
        recurse(undertaker, undertaker_score)  # 递归进入下一层

function recurse(undertaker_node, allocated_score):
    if is_person(undertaker_node):
        # 到达个人层，记录得分
        person_scores[undertaker_node.id] += allocated_score
    else:
        # 仍是组织层，继续向下
        child_bps = get_bps(undertaker_node)
        child_bp_scores = ai_score_peers(child_bps)  # 同级评分，总和=allocated_score
        for child_bp, child_score in zip(child_bps, child_bp_scores):
            distribute(child_bp, child_score)
```

#### taskDepts 承接关系处理规则（补充说明，修复 BLK-02）

**规则一：举措粒度独立处理**
同一BP的不同举措可以由不同部门/人承接，每条举措的承接方独立处理。不将整个BP视为一个承接单元，而是将举措作为最小分配粒度：
```
for measure in bp.measures:
    undertakers = [dept for dept in measure.taskDepts if dept.role == "undertaker"]
    distribute_measure_score(measure, undertakers)
```

**规则二：deptId → groupId 跳转**
`taskDepts`返回的是`deptId`（部门标识），但递归需要进入该部门的BP树，必须先获取`groupId`：
```
# 用 API 4.12 按部门名称模糊搜索，获取对应 groupId
POST /bp/group/searchByNameFuzzy
Body: { "name": dept.name, "type": "dept" }
→ 返回匹配的 groupId
→ 再用 API 4.4 GET /bp/group/getTree?groupId={groupId} 获取BP树
```
若搜索返回多个结果，按完整名称精确匹配优先，仍有歧义则标注`[承接方定位模糊]`，记入待人工确认项。

**规则三：同一举措多个承接方的分值分配**
当一条举措的`taskDepts`中有多个`role=undertaker`时，这多个承接方共同分享该举措对应的分值池：
```
if len(undertakers) > 1:
    # AI评估各承接方的贡献度比例，比例之和 = 1.0
    ratios = ai_score_undertakers(measure, undertakers)
    for undertaker, ratio in zip(undertakers, ratios):
        undertaker_score = measure_score * ratio
        recurse(undertaker, undertaker_score)
```

### 3.4 边界情况处理

| 情况 | 处理方式 | 报告标注 |
|---|---|---|
| 某BP **无任何承接方** | 分值保留在当前层，不往下分配 | `[无承接]` |
| 某BP **只有1个承接方** | 承接方获得100%分值池，无需AI比较 | `[独家承接]` |
| 承接方 **BP内容为空/极简** | AI按最低贡献度处理（≥0.5分），标注 | `[内容不完整]` |
| **某人承接多个BP** | 其最终得分 = 所有承接BP得分之和，分别列出来源 | 汇总表中列出各来源 |
| **组织名搜索到多个结果** | 列出所有匹配项，要求用户选择，不自动猜测 | 交互提示 |
| **AI置信度 < 阈值** | 列入"待人工确认项"，用户可选择接受或调整 | `[低置信度]` |
| **分值计算出现浮点误差** | 四舍五入到小数点后1位，保证总分=100 | 无 |
| **某层级BP数量 > 20** | 分批调用AI（每批≤10个），合并结果后归一化 | `[分批处理]` |
| **LLM调用失败** | 重试3次，失败后标注为人工确认项，不中断流程 | `[AI评分失败，需人工]` |
| **BP内容质量极差**（目标空洞/KR无量化/举措缺失） | 不过滤，正常参与评分，但AI在评分理由中明确说明内容缺陷，分值会相应偏低（可低至最低分1分）；报告中标注`[内容质量低]`并列出具体缺陷，确保用户理解低分原因，而非看到低分却不知为何 | `[内容质量低：KR无量化/举措缺失]` |
| **完成时间节点偏差**（某BP的举措承诺Q1完成但已过期未交付） | 在AI评分的「举措覆盖度」和「成果对上级贡献」维度中扣分，理由中明确写明"已超预期完成时间节点"；若时间数据无法从BP文本中提取，则不扣分，标注`[时间节点无法判断]` | `[时间节点已过期]` |

### 6.2 调整规则

#### 「原始比例」基准定义（修复 BLK-03）

- **"原始比例"定义为：本次会话中AI首次生成的比例**，在评分完成后立即存入 session 状态（内存/临时文件），格式为：
  ```json
  {
    "layer_id": "产品中心-2026Q1",
    "original_ratios": {
      "BP-001": 0.28,
      "BP-002": 0.22,
      "BP-003": 0.20,
      "BP-004": 0.18,
      "BP-005": 0.12
    }
  }
  ```
- 原始比例在整个会话期间不变，每次 adjust 后**不更新**原始比例基准

#### 调整联动计算规则

- 调整某BP分值后，**从原始AI比例**重新计算同层其他BP的剩余分配（等比缩放）：
  ```
  已调整BP得分 = 用户指定值（如70）
  剩余分 = 100 - 70 = 30
  其他BP的原始比例之和 = sum(original_ratios[bp] for bp in 未调整BPs)
  其他BP的新得分 = 剩余分 × (original_ratios[bp] / 其他BP原始比例之和)
  ```
  **示例**：原始 A=50, B=30, C=20。调整A→70，剩余30分由B、C按原始比例3:2分配，B=18，C=12。若再调整B→25，剩余5分归C，C=5（B、C原始比例2:1，但B已被用户指定，C直接获得余量）。
  
  > **说明**：当同层中已有多个BP被人工调整时，剩余分由"尚未被人工指定的BP"按其原始比例等比瓜分；全部被手动指定时，若总和≠100则报错，要求用户修正。

- 调整某层级BP后，**其下级的分值池自动联动更新**（递归重算）
- 每次调整必须填写 `--reason`，原因记录在报告中

#### 报告输出标注

评分报告中每个BP分值旁必须明确标注来源：
- `[AI原始]`：未经人工调整，直接使用AI评分结果
- `[人工调整]`：已由人工通过 adjust 命令覆盖，调整原因跟随显示

示例：
```
- BP-001《构建核心产品竞争力》— 70.0分 [人工调整，原AI评分28.0分，原因：战略重要性被AI低估]
- BP-002《提升产品交付效率》— 18.0分 [AI原始]
- BP-003《组织能力建设》— 12.0分 [AI原始，联动更新自BP-001调整]
```

#### 其他约束

- 已confirm的报告不可再修改（需创建新版本）
- Session结束后原始比例不持久化（每次 `run` 命令重新生成新的原始比例基准）

## 七、迭代机制

所有评分维度和权重存放在 `config/scoring_weights.yaml`，**修改此文件无需改任何代码**：

```yaml

pool_distribution:
  target_accuracy: 0.45         # 目标承接准确性
  outcome_contribution: 0.35    # 成果对上级贡献
  measure_completeness: 0.20    # 举措完整性

system:
  confidence_threshold: 0.70    # AI置信度阈值（低于此值列入待确认项）

## 十一、奖金系数合理性评价

### 11.1 背景与定位

API 4.17（`batchGetKeyPositionMarkdown`）返回的奖金系数建议是**人工设定**的，反映了管理者的主观判断。本系统的AI评分结果基于BP内容的客观分析，两者各有侧重：

- **AI评分**：基于BP文本的战略对齐度、可衡量性、举措覆盖度等维度，客观量化各人的相对贡献度
- **系统系数建议**（API 4.17）：基于岗位属性和管理者经验设定，可能包含AI无法感知的隐性信息

两者关系：**互为参考，不是替代**。本节提供对照评价，辅助管理者发现潜在的不一致，最终决策权在管理者。

### 11.2 评价维度

#### 维度一：纵向对比（AI评分 vs 人工系数建议）

将AI计算出的个人分值（百分制，代表在本组织中的相对贡献度占比）与API 4.17返回的人工奖金系数建议进行比对：

| 评价结论 | 判定逻辑 |
|---|---|
| ✅ 合理 | AI排名与人工系数排名差异 ≤ 1位，或分值差异 < 15% |
| ⚠️ 偏高 | 人工系数建议排名 显著高于 AI评分排名（差异 ≥ 2位，且分值差 ≥ 15%） |
| ⚠️ 偏低 | 人工系数建议排名 显著低于 AI评分排名（差异 ≥ 2位，且分值差 ≥ 15%） |
| ❓ 无法比较 | 该员工标注`[未建BP]`，或AI评分置信度 < 0.60 |

**差异百分比计算方式**：
```
ai_rank_pct = (ai_score / total_score) × 100%          # AI分值占比
human_rank_pct = 该员工系数 / 同组织所有人系数之和 × 100%  # 人工系数占比
diff_pct = abs(ai_rank_pct - human_rank_pct)
```

#### 维度二：横向对标（同组织排名 + 跨组织同岗位）

**同组织内排名对比**：
- 将所有员工按AI分值排名（降序）
- 将所有员工按人工系数建议排名（降序）
- 对比两个排名，找出排名差异最大的人员，重点标注

**跨组织同岗位对标**：
- 通过API 4.17返回的岗位title，收集其他组织中相同/相近岗位的员工数据
- 横向比较：AI分值分布 vs 人工系数分布，判断该岗位在全公司范围内的系数设置是否一致

```
同岗位对标逻辑：
1. 从当前组织所有员工的API 4.17响应中提取岗位title
2. 对其他已评分组织中，找到相同title的员工
3. 输出：同岗位人员的AI分值范围 + 人工系数范围 + 是否存在跨组织差异
```

### 11.5 实现要点

- `bonus_checker.py` 在 `main.py confirm` 命令执行时**自动触发**，不需要单独命令

跨组织对标需要读取其他已评分组织的结果。缓存结构如下：

**缓存文件路径**：`output/.cross_org_cache.json`（隐藏文件，自动维护）

**缓存结构**：
```json
{
  "version": 1,
  "last_updated": "2026-03-29T20:00:00",
  "orgs": {
    "产品中心": {
      "confirmed_at": "2026-03-29T20:00:00",
      "persons": [

          "name": "员工甲",
          "title": "产品经理",

      ]
    }
  }
}

**读写时机**：
- **写入**：每次 `confirm` 命令成功执行后，自动将当前组织的个人评分结果写入缓存
- **读取**：`bonus_checker.py` 运行时读取缓存，提取同岗位人员数据用于横向对标

- **过期策略**：缓存无过期机制（BP周期内数据稳定），手动 `python3 scripts/main.py cache clear` 可清除
