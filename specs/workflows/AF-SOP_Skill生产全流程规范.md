# AF-SOP Skill 生产全流程规范

- 版本：v5.2
- 状态：STABLE
- 合并自：AF-SOP-01（v4.7）+ AF-SOP-02（v1.1）+ SKILL-QUALITY-PRINCIPLES（v1.2）
- 合并日期：2026-04-19
- 适用范围：工厂所有 Skill 从需求到交付的全生命周期

---

## 0. 工厂定位

Agent Factory 不是一个项目，而是一条 Skill 产品生产线。

---

## 1. 核心角色

| 角色 | 核心职责 | 三省对应 |
|---|---|---|
| 工厂调度员 (Orchestrator) | 产品经理、调度决策、用户窗口 | — |
| 设计总工 (Chief Designer) | 出图纸、定标准、方案架构设计 | 中书省 |
| 交付总管 (Delivery Manager) | 生产执行、代码编写、打包发布 | 尚书省 |
| 质检总监 (Quality Director) | 找茬压测、红蓝对抗、质量把控 | 门下省 |

---

## 2. 生命周期

```
S0 需求接收 → S1 需求澄清 → S2 需求固化 → S3 方案设计（S3a 认知扫描 + S3b 方案生成） → S4 开发实现 → S5 质量验证 → S6 发布 → S7 交付确认 → S8 归档沉淀
```

| 阶段 | 核心任务 | 责任主体 | 关键产出 |
|---|---|---|---|
| S0 需求接收 | 确认是否是 Skill 需求 | 调度员 | 快速筛选结论 |
| S1 需求澄清 | 5W1H 深度访谈 | 调度员 | 口头确认 |
| S2 需求固化 | 输出 REQ-01 | 调度员 | `REQ-01.md` |
| S3 方案设计 | 认知扫描（可选）+ 技术方案、数据流、接口规范 | 设计总工 | `step3-domain-scan.md` + `DESIGN-01.md` |
| S4 开发实现 | 编写代码、SKILL.md、脚本 | 交付总管 | `CODE-01.md` |
| S5 质量验证 | 功能测试 + 10 维度审计 | 质检总监 | `TEST-01.md` |
| S6 发布 | 打包到 releases/ | 交付总管 | `releases/*.zip` |
| S7 交付确认 | 用户验收 | 调度员 | 验收确认 |
| S8 归档沉淀 | 更新设计档案、经验记录 | 调度员 | 档案更新 |

每步完成后必须用户确认，才能进入下一步。

---

## 3. S0: 需求接收

### 快速筛选

**不是 Skill（拒绝）**：
- 一次性任务 → 直接执行
- 已有 Skill 可覆盖 → 复用或微调
- 需要探索多文件 → 用 subagent
- "为了建 Skill 而建 Skill" → 拒绝

**是 Skill（继续）**：可复用、有特定工作流、需要进阶披露。

---

## 4. S1: 需求澄清

5W1H 深度访谈，不要一次问完所有问题：
1. 先听用户完整描述
2. 针对模糊点追问（每次最多 3 个问题）
3. 用用户的话术反馈理解，确认无误

**必须明确**：用户画像、核心场景、触发词、核心功能（1-3 个）、不做什么。

---

## 5. S2: 需求固化

输出 `REQ-01.md`，包含：
- 用户画像（使用者、使用频率、使用场景）
- 核心功能列表
- 触发词
- 不做什么（边界）
- 约束条件
- 验收标准

向用户展示摘要，请求确认后进入 S3。

---

## 6. S3: 方案设计

### 6.0 主动提示规则（认知扫描）

**触发时机**：Orchestrator 在以下场景应主动向用户建议执行认知扫描——

1. **S2→S3 过渡时**：当需求涉及一个有成熟方法论体系的业务领域（如销售、客服、培训、财务），主动提示：
   > "这个需求涉及 [领域名]，该领域有成熟的专家共识和不同的方法论流派。建议在方案设计前做一次**领域认知扫描**，帮你理清：哪些是必须覆盖的共识能力，哪些是需要你做立场选择的分歧点。要做吗？"

2. **S3 方案讨论中**：当用户在设计框架/模板/关键章节时出现犹豫或反复，主动提示：
   > "看起来我们在框架设计上还在探索。要不要先做一个领域认知扫描？把专家共识和分歧梳理清楚，框架自然就出来了。"

3. **用户直接提及领域术语**：如"最佳实践"、"行业标准"、"方法论"等词，主动提示：
   > "你提到了 [术语]，说明这个领域有成熟的认知体系。建议做一次认知扫描，确保我们的方案不遗漏关键共识，同时对分歧点做出明确选择。"

**用户可以拒绝**：如果用户明确表示"不需要"，直接进入 6.2。不强制。

### 6.1 S3a: 认知扫描（可选，建议执行）

**责任主体**：设计总工（调度 Analyst Agent 执行）

**输入**：S2 产出的 REQ-01.md + 用户口头的领域背景

**执行内容**：
1. 识别该领域的核心专家共识（最多5个心智模型）
2. 识别该领域的根本性分歧（最多3个争议点）
3. 将共识映射为方案的必选章节
4. 将分歧映射为需要用户做立场选择的设计决策点

**输出**：`step3-domain-scan.md`

**向用户展示**：扫描摘要，重点展示分歧点和选项，请用户做立场选择。

**用户确认后**，带着立场选择结果进入 S3b。

### 6.2 S3b: 方案生成

#### 6.2.1 设计总工负责
1. 技术方案（架构、技术栈）
2. 数据流（来源、处理、输出）
3. 接口规范（涉及哪些 API）
4. 文件结构（最小结构）
5. 用户体验（失败兜底、错误处理）
6. **如果存在认知扫描结果**：按扫描产出的章节映射生成方案，分歧点按用户选择的立场写入

#### 6.2.2 粒度判断

**拆分**：不同权限集 / 可独立完成 / 触发条件不同 / 完成超 5 分钟
**合并**：步骤不可分割 / 共享权限和上下文

#### 6.2.3 唯一真相源（涉及外部 API 时强制）

1. 用 curl 获取官方文档，不凭记忆推断
2. 禁止使用本地缓存文档作为权威来源
3. DESIGN-01 必须包含官方文档链接和一致性声明

#### 6.2.4 输出 DESIGN-01

包含：粒度决策、技术方案、数据流、API 规范（唯一真相源）、文件结构、用户体验、**认知扫描引用（如有）**。

向用户展示摘要，请求确认后进入 S4。
## 7. S4: 开发实现

### 7.1 交付总管负责
1. 按 DESIGN-01 创建目录和文件
2. 编写核心代码
3. 编写 SKILL.md
4. 编写测试用例

### 7.2 三层披露（Token 经济学）

| Tier | 内容 | 加载时机 | 预算 |
|------|------|----------|------|
| 1 | frontmatter description | 每次对话都加载 | ~100 词 |
| 2 | SKILL.md 正文 | Skill 触发后加载 | **80 行以内** |
| 3 | references/ 辅助文件 | 按需加载 | 不限 |

细节只能向下移，不能向上堆。

### 7.3 description 设计

- 必须包含 **3-5 个具体触发词**，覆盖口语化表达
- 用动词开头，不以 "Use when..." 开头
- 要"有点激进"，LLM 默认不触发 Skill

### 7.4 写法规范

命令式 + 解释原因。不要用大写强调（ALWAYS/NEVER），要说清楚"为什么"。

❌ "ALWAYS 先存草稿，NEVER 直接发送"
✅ "先存草稿再发送，因为直接发没有确认机会，出错无法撤回"

### 7.5 涉及外部 API 的 SKILL.md

必须包含官方文档链接，不维护本地 API 端点表。

### 7.6 最小结构

只有 SKILL.md，够用就停。references/ ≤ 3 个文件，scripts/ 只在确实需要时才存在。

向用户展示 CODE-01 摘要，请求确认后进入 S5。

## 8. S5: 质量验证

### 8.1 构建测试包

```bash
cd projects/{id}/{skill-name}
zip -r ../builds/v{版本号}-$(date +%Y%m%d-%H%M).zip . \
  -x '*.pyc' -x '__pycache__/*' -x '*.DS_Store' -x 'builds/*' -x 'releases/*'
```

### 8.2 自动检查（我做完）

构建完测试包后，我自动执行以下检查并汇报结果：

- SKILL.md 行数是否 ≤ 80 行
- frontmatter 是否有 name、description、version
- description 是否含 3-5 个触发词
- 引用的文件路径是否全部存在
- 有无凭证泄露（token、password、secret）
- references/ 文件是否 ≤ 3 个
- 有无重复内容（SKILL.md 和 references/ 之间）

### 8.3 测试计划（发给 Evan）

自动检查完成后，我生成一份简短的测试计划发送给你：

```
📋 {skill-name} 测试计划

✅ 自动检查：[已通过的项]
❌ 自动检查：[未通过的项 + 修复建议]

请你在对话中测试以下场景：
1. 正常触发：说「{触发词}」，验证 {预期行为}
2. 边界场景：说「{边界输入}」，验证 {预期处理}
3. 不应触发：说「{相似请求}」，验证不会误触发

测试完成后回复 结果 或 问题。
```

你测试完告诉我结果，有问题我修，没问题进入 S6 发布。

用户确认功能符合需求、体验良好、可以开始使用。

## 11. S8: 归档沉淀

1. 更新设计档案（DESIGN.md、DISCUSSION-LOG.md）
2. 记录经验教训到 LEARNING-LOOP.md
3. 更新全局项目索引

---

## 12. 设计档案规范

每个 Skill 目录下必须维护 `design/` 档案集：

- `DESIGN.md` — 产品目标、边界、核心流程、关键决策
- `DISCUSSION-LOG.md` — 每次讨论的要点、结论、待办
- `LEARNING-LOOP.md` — 复盘框架、改进建议
- `SHARE-LOG.jsonl` — 经验分享去重台账

**讨论后强制维护**：
1. 更新 DISCUSSION-LOG.md
2. 有设计变更 → 更新 DESIGN.md
3. 有改进建议 → 更新 LEARNING-LOOP.md
4. 有对外分享 → 记录 SHARE-LOG.jsonl

未完成以上动作，不得标记该轮讨论"已闭环"。

---

## 13. 复盘与分享机制

**手动触发**：用户通过自然语言触发（"生成今日复盘"、"分享未分享的日经验"等）。

**去重规则**：已分享内容不重复，强制分享需用户明确要求。每次分享前查询 SHARE-LOG.jsonl。

**复盘必须产出**：
- 问题修复沉淀（Problem → Rule）
- 预期偏差校正（Mismatch → Preference）
- 用户工作模式更新（Pattern → Twin）

---

## 14. 颗粒度决策树

```
收到新需求
    ├─ 一次性任务？ → 直接执行，不建 Skill
    ├─ 已有 Skill 可覆盖？ → 复用或微调，不新建
    ├─ 需要探索多文件？ → 用 subagent，不建 Skill
    └─ 可复用、有特定工作流、需要进阶披露？
           → 建 Skill，从最小结构开始
```
