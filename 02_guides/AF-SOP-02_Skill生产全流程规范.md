## 0. 核心理念

**Skill 工厂不是项目，而是一条生产线。**

用户输入 → 需求确认 → 方案设计 → 开发实现 → 质量验证 → 四重发布 → 持续维护

## 2. S0: 需求接收（Trigger）

### 2.1 触发方式

用户通过自然语言提出 Skill 需求，例如：

- "我想做一个 Skill，可以自动发送工作汇报"
- "需要一个 BP 管理工具，能查看目标和 KR"
- "能不能帮我做一个会议监控 Skill？"

### 2.2 工厂调度员的第一步

**立即执行**：
1. 确认是否真的是 Skill 需求（不是一次性任务、不是 subagent 任务）
2. 快速判断是否已有 Skill 可以覆盖
3. 开始需求澄清

### 2.3 快速筛选

**不是 Skill 的情况（拒绝）**：
- 一次性任务：直接执行，不建 Skill
- 已有 Skill 可以覆盖：复用或微调，不新建
- 工作量大需要探索多文件：用 subagent，不建 Skill
- "为了建 Skill 而建 Skill"：拒绝

**是 Skill 的情况（继续）**：
- 可复用的需求
- 有特定工作流
- 需要进阶披露（Tier 2/3）

---

## 3. S1: 需求澄清（Clarification）

### 3.1 5W1H 深度访谈

工厂调度员需要回答 6 个核心问题：

| 问题 | 为什么问 | 示例 |
|------|---------|------|
| **Who** | 谁会使用这个 Skill？ | "是给管理者用，还是普通员工用？" |
| **What** | Skill 具体要做什么？ | "是要查看 BP，还是要编辑 BP？" |
| **When** | 什么时候触发？ | "用户说什么词时触发？" |
| **Where** | 在什么场景下使用？ | "是在写周报时，还是在开会时？" |
| **Why** | 为什么要建这个 Skill？ | "是为了节省时间，还是为了确保准确性？" |
| **How** | 大概怎么实现？ | "需要调用哪些 API？有哪些关键步骤？" |

### 3.2 交互方式

**不要一次性问完所有问题**，而是：
1. 先听用户完整描述
2. 针对模糊点追问
3. 用用户的话术反馈理解，确认无误

**示例对话**：

```
用户：我想要一个 BP 管理工具。

调度员：好的，BP 管理工具。想确认一下：
1. 是要查看自己的 BP，还是也要能看下级的？
2. 是只查看，还是需要能新增目标和 KR？
3. 你希望用什么触发词？比如"BP管理"、"查看目标"？

用户：1. 都要看，2. 先只查看，3. "BP管理"就行。

调度员：明白了。那你的主要场景是：
- 作为管理者，快速查看自己与下级的 BP 全貌
- 目前不需要编辑功能，只读查看
- 用"BP管理"这个触发词

是这个意思吗？

用户：对。
```

### 3.3 需求澄清完成的标准

**必须明确**：
- 用户画像（谁用）
- 核心场景（在什么情况下用）
- 触发词（用户说什么词触发）
- 核心功能（至少 1-3 个）
- 不做什么（边界）

**输出**：口头确认，用户回复"对"或"是的"。

## 5. S3: 方案设计（Design）

### 5.1 设计总工的任务

设计总工负责：
1. **技术方案**：如何实现（架构、技术栈）
2. **数据流**：数据从哪里来，到哪里去
3. **接口规范**：涉及哪些 API，如何调用
4. **文件结构**：目录结构、文件组织
5. **用户体验**：失败兜底、错误处理

### 5.2 粒度判断（设计原则 #4）

**应该拆分的情况**：
- 需要不同权限集
- 可以独立完成，不互相依赖
- 触发条件根本不同
- 完成时间超过 5 分钟

**应该合并的情况**：
- 步骤不可分割
- 共享完全相同的权限和上下文

**粒度决策树**：

```
收到新需求
    │
    ├─ 一次性任务？ → 直接执行，不建 Skill
    │
    ├─ 已有 Skill 可以覆盖？ → 复用或微调，不新建
    │
    ├─ 工作量大，需要探索多文件？ → 用 subagent，不建 Skill
    │
    └─ 可复用、有特定工作流、需要进阶披露？
           → 建 Skill，从最小结构开始
```

**最小结构原则**：只有 `SKILL.md`，够用就停。不要过度设计。

### 5.3 唯一真相源验证（设计原则 #7）

**如果 Skill 涉及外部 API**，设计总工必须：

1. **用 curl 获取官方文档**：
   ```bash
   curl -sL "https://github.com/xgjk/dev-guide/raw/main/..."
   ```

2. **验证 API 规范**：
   - 接口路径、方法、参数、响应格式
   - 与代码实现（如果已有）是否一致

3. **输出 DESIGN-01** 时，必须包含：
   - 官方文档链接
   - 关键 API 摘要
   - 代码实现与官方文档的一致性声明

**反面案例（必须避免）**：
- 凭记忆推断 API 规范
- 使用本地 `api-endpoints.md` 作为权威来源
- 使用 web_search/web_fetch 获取文档（可能被阻止）
- 因为"代码已经这样写"而拒绝修改以符合官方规范

### 5.4 输出 DESIGN-01

```markdown
# DESIGN-01 - [Skill名称] 设计文档

## 粒度决策
- [ ] 为什么需要建 Skill（不是一次性任务/subagent/复用现有）
- [ ] 触发条件：[描述]
- [ ] 拆分/合并理由：[描述]

## 技术方案
- 技术栈：[Python/Node.js/其他]
- 架构：[架构图或描述]
- 核心依赖：[依赖列表]

## 数据流
- 数据来源：[API/文件/数据库]
- 数据处理：[处理逻辑]
- 数据输出：[输出格式]

## API 规范（唯一真相源）
- 涉及的 API 列表
- 官方文档链接
- 关键接口摘要
- 代码实现与官方文档的一致性声明

## 文件结构（最小结构）
```
skill-name/
├── SKILL.md
├── scripts/
│   ├── main.py
│   └── utils.py
├── references/
│   └── maintenance.md
└── design/
    ├── DESIGN.md
    └── DISCUSSION-LOG.md
```

## 用户体验
- 正常流程：[步骤]
- 异常处理：[失败兜底]
- 错误提示：[用户友好的错误信息]

## 质量标准
- 响应时间：[要求]
- 准确性：[要求]
- 可用性：[要求]
```

### 5.5 用户确认

**必须**：
1. 向用户展示 DESIGN-01 摘要
2. 请求确认："设计方案是这样，确认后我开始开发。"
3. 用户回复"确认"或"是的"后，进入 S4

## 6. S4: 开发实现（Development）

### 6.1 交付总管的任务

交付总管负责：
1. **创建文件结构**：按 DESIGN-01 创建目录和文件
2. **编写核心代码**：实现核心功能
3. **编写 SKILL.md**：遵循三层披露原则
4. **编写测试用例**：正常、边界、异常场景

### 6.2 三层披露原则（设计原则 #2，Token 经济学）

**核心思想**：细节只能向下移，不能向上堆。

| Tier | 内容 | 加载时机 | 预算 |
|------|------|----------|------|
| 1 | frontmatter description | 每次对话都加载 | ~100 词 |
| 2 | SKILL.md 正文 | Skill 触发后加载 | **80 行以内** |
| 3 | references/ 辅助文件 | 按需加载，SKILL.md 指引 | 不限 |

**实施规则**：
- frontmatter 的 description 是 AI 每次对话都看到的，必须精炼
- SKILL.md 正文是 Skill 触发后加载的，控制在 80 行以内
- 细节（如 API 端点列表、复杂配置）推入 references/

### 6.3 触发词设计（设计原则 #3）

**description 要"有点激进"**：
- LLM 默认不触发 Skill，宁可自己回答
- description 必须包含 **3-5 个具体触发词**，覆盖口语化表达
- 可以加 "even if they don't say XXX" 强制触发
- **不要以 "Use when..." 开头**，改用动词开头

**反面案例**：
```
❌ description: "Use when the user wants to search for employees."
✅ description: "搜索员工。触发词：找人、查员工、搜一下谁、查查这个名字"
```

**正面案例**：
```
✅ description: "发送工作汇报。触发词：发周报、发汇报、提交日报、周报发出去了吗（即使用户没直接说'发送'）"
```

### 6.4 写法规范（设计原则 #5）

**命令式 + 解释原因**：

| 错误写法 | 正确写法 |
|---------|---------|
| "ALWAYS 先存草稿，NEVER 直接发送" | "先存草稿再发送，因为直接发没有确认机会，出错无法撤回" |
| "CRITICAL: 必须设置超时" | "设置超时，因为 API 响应慢时会导致会话卡住" |

**核心原则**：LLM 理解原因比遵守规则更可靠。不要用大写强调，要解释为什么。

### 6.5 SKILL.md 编写规范

**frontmatter 必须包含**：
```yaml
---
name: skill-name
description: "[3-5 个具体触发词，覆盖口语化表达]"
homepage: https://github.com/evan-zhang/agent-factory
issues: https://github.com/evan-zhang/agent-factory/issues/new?labels=skill-name
version: "1.0.0"
tools_provided:
  - name: tool1
    category: exec
    risk_level: medium
    permission: exec
    description: "[工具描述]"
    status: active
metadata:
  requires:
    env:
      - ENV_VAR
---
```

**涉及外部 API 的 Skill，SKILL.md 必须包含**：

```markdown
## API 文档

> ⚠️ **重要提示**：本 Skill 调用玄关开放平台 API，权威文档是玄关开放平台官方文档。
>
> **官方文档链接**：https://github.com/xgjk/dev-guide/...
>
> 本 Skill 的代码实现与官方文档完全一致。如需查看完整的 API 规范，请直接访问上述官方文档。
```

**SKILL.md 纪律**：
- [ ] ≤ 80 行，容易扫描
- [ ] 只包含核心工作流和规则
- [ ] 细节已推入 references/
- [ ] 无废话段落（如使命宣言）
- [ ] 引用的辅助文件路径明确列出

### 6.6 代码实现规范

**必须遵循**：
1. **唯一真相源原则**：代码实现与官方文档完全一致
2. **错误处理**：所有 API 调用都有 try-except
3. **日志输出**：关键步骤有日志输出
4. **环境变量**：敏感信息通过环境变量传递
5. **类型提示**：Python 代码使用类型提示

### 6.7 最小结构原则（设计原则 #5）

**目录结构是能满足需求的最小结构**：
- references/ 文件 ≤ 3 个
- assets/ 只在需要捆绑静态文件时才存在
- scripts/ 只在确实需要确定性执行时才存在

## 7. S5: 质量验证（Testing）

### 7.1 质检总监的任务

质检总监负责：
1. **功能测试**：正常场景、边界场景、异常场景
2. **质量审计**：10 维度 Audit Checklist
3. **安全检查**：凭证、敏感信息、扫描触发词
4. **文档检查**：SKILL.md、references/ 完整性

### 7.2 Smoke Test（设计原则 #6）

**每个 Skill 发布前必须过三类测试**：

1. **正常触发**：标准场景，应该触发且行为正确
2. **边界触发**：边缘场景，应该触发并能处理
3. **不应触发**：相似但不属于该 Skill 的请求，不应触发

**示例**（假设 Skill 是"发送工作汇报"）：
- 正常触发："帮我发一个周报"
- 边界触发："发个汇报，但是内容有点长"（处理长内容）
- 不应触发："帮我搜索一下员工"（不应触发，应触发搜索员工的 Skill）

### 7.3 10 维度 Audit Checklist

**必须全部通过才能进入 S6**：

#### 维度 1 — 触发质量
- [ ] description 说清楚了做什么、什么时候用？
- [ ] 含 3-5 个具体触发词，覆盖口语化表达？
- [ ] 用动词开头，不以 "Use when..." 开头？
- [ ] 和其他已有 Skill 没有明显重叠？
- [ ] 描述足够"激进"，能在用户不说关键词时也能触发？

#### 维度 2 — 是否真的需要建 Skill
- [ ] 这是可复用的需求，不是一次性任务？
- [ ] 没有现成工具或 Skill 可以直接覆盖？
- [ ] 不是 subagent 任务（工作量大、需要探索多文件）？
- [ ] 不是"为了建 Skill 而建 Skill"？

#### 维度 3 — SKILL.md 纪律
- [ ] SKILL.md ≤ 80 行，容易扫描？
- [ ] 只包含核心工作流和规则，细节已推入 references/？
- [ ] 引用的辅助文件路径明确列出，可发现？
- [ ] 没有废话段落？

#### 维度 4 — 写法质量
- [ ] 命令式语气（"读取文件" 不是 "可以考虑读取"）？
- [ ] 关键步骤解释了"为什么"，不只是"是什么"？
- [ ] ALWAYS/NEVER/CRITICAL 大写附带了原因解释？
- [ ] 示例 1-2 个，简洁具体，不堆砌？

#### 维度 5 — 最小化原则
- [ ] 目录是能满足需求的最小结构？
- [ ] references/ 文件 ≤ 3 个？
- [ ] assets/ 只在需要捆绑静态文件时才存在？
- [ ] scripts/ 只在确实需要确定性执行时才存在？

#### 维度 6 — 文件卫生
- [ ] 无 README/CHANGELOG 等对 AI 无用的人类文档？
- [ ] 无 SKILL.md 和 references/ 之间的重复内容？
- [ ] 无废弃的占位文件或实验残留？

#### 维度 7 — 规范漂移检查
- [ ] 引用的文件路径真实存在？
- [ ] 示例代码/命令仍然准确有效？
- [ ] Skill 没有悄悄扩展到多个目的（"越界"）？
- [ ] 指令与当前工具/工作流仍然匹配？

#### 维度 8 — Smoke Test（三场景必过）
- [ ] 正常触发：标准场景，触发且行为正确？
- [ ] 边界触发：边缘场景，触发且能处理？
- [ ] 不应触发：相似但不属于该 Skill 的请求，不触发？

#### 维度 9 — Do-nothing Check（防过度工程）
- [ ] 改一个文件能解决的，没有错误地建 Skill？
- [ ] 这不是 subagent 问题（而非 Skill 问题）？
- [ ] 新增内容真的在承担自己的 token 成本？

#### 维度 10 — 安全与元数据（工厂专项）
- [ ] 无 process.env 写入或读取凭证（用 runtime 模块替代）？
- [ ] External Endpoints 在 setup.md 中明确声明？
- [ ] frontmatter metadata.requires.env 已声明必需变量？
- [ ] 无 "silently/secretly/automatically/monitor/track" 等扫描触发词？
- [ ] LLM 凭证由调用方注入，Skill 内部不存储？
- [ ] references/maintenance.md 存在且包含 GitHub 地址和提 issue 入口？

### 7.4 唯一真相源验证（设计原则 #7，强制）

**如果 Skill 涉及外部 API**，质检总监必须：

1. **用 curl 获取最新官方文档**：
   ```bash
   curl -sL "https://github.com/xgjk/dev-guide/raw/main/..."
   ```

2. **对比代码实现**：
   - 接口路径、方法、参数、响应格式是否一致
   - 如有差异，**以官方文档为准，拒绝通过**

3. **检查 SKILL.md**：
   - 是否指向官方文档链接
   - 是否有本地 API 端点表（如有，要求删除）

## 11. 十大设计原则速查

| # | 原则 | 核心要点 | 融入阶段 |
|---|------|---------|---------|
| **#1** | 单一职责 | 一个 Skill 只做一件事，边界清晰 | S3 粒度判断 |
| **#2** | 三层披露 | Token 经济学，细节向下移 | S4 SKILL.md 编写 |
| **#3** | 触发词激进 | 3-5 个触发词，用动词开头 | S4 description 设计 |
| **#4** | 粒度判断 | 拆分/合并的判断标准 | S3 方案设计 |
| **#5** | 写法规范 | 命令式 + 解释原因 | S4 文档编写 |
| **#6** | Smoke Test | 三场景测试 | S5 测试验证 |
| **#7** | 唯一真相源 | 官方文档是唯一权威来源 | S3/S5 API 验证 |
| **#8** | 最小结构 | 满足需求的最小目录结构 | S4 文件组织 |
| **#9** | 防过度工程 | 改一个文件能解决的不建 Skill | S0/S3 筛选 |
| **#10** | 安全元数据 | 凭证、endpoint、issue 入口 | S5 安全检查 |
