## 0. 核心理念

**Skill 工厂不是项目，而是一条生产线。**

用户输入 → 需求确认 → 方案设计 → 开发实现 → 质量验证 → 四重发布 → 持续维护

## 2. S0: 需求接收（Trigger）

### 2.1 触发方式

用户通过自然语言提出 Skill 需求，例如：

- "我想做一个 Skill，可以自动发送工作汇报"
- "需要一个 BP 管理工具，能查看目标和 KR"
- "能不能帮我做一个会议监控 Skill？"

### 2.2 工厂调度员的第一步

**立即执行**：
1. 确认是否真的是 Skill 需求（不是一次性任务、不是 subagent 任务）
2. 快速判断是否已有 Skill 可以覆盖
3. 开始需求澄清

### 2.3 快速筛选

**不是 Skill 的情况（拒绝）**：
- 一次性任务：直接执行，不建 Skill
- 已有 Skill 可以覆盖：复用或微调，不新建
- 工作量大需要探索多文件：用 subagent，不建 Skill
- "为了建 Skill 而建 Skill"：拒绝

**是 Skill 的情况（继续）**：
- 可复用的需求
- 有特定工作流
- 需要进阶披露（Tier 2/3）

## 3. S1: 需求澄清（Clarification）

### 3.1 5W1H 深度访谈

工厂调度员需要回答 6 个核心问题：

| 问题 | 为什么问 | 示例 |
|------|---------|------|
| **Who** | 谁会使用这个 Skill？ | "是给管理者用，还是普通员工用？" |
| **What** | Skill 具体要做什么？ | "是要查看 BP，还是要编辑 BP？" |
| **When** | 什么时候触发？ | "用户说什么词时触发？" |
| **Where** | 在什么场景下使用？ | "是在写周报时，还是在开会时？" |
| **Why** | 为什么要建这个 Skill？ | "是为了节省时间，还是为了确保准确性？" |
| **How** | 大概怎么实现？ | "需要调用哪些 API？有哪些关键步骤？" |

### 3.2 交互方式

**不要一次性问完所有问题**，而是：
1. 先听用户完整描述
2. 针对模糊点追问
3. 用用户的话术反馈理解，确认无误

**示例对话**：

```
用户：我想要一个 BP 管理工具。

调度员：好的，BP 管理工具。想确认一下：
1. 是要查看自己的 BP，还是也要能看下级的？
2. 是只查看，还是需要能新增目标和 KR？
3. 你希望用什么触发词？比如"BP管理"、"查看目标"？

用户：1. 都要看，2. 先只查看，3. "BP管理"就行。

调度员：明白了。那你的主要场景是：
- 作为管理者，快速查看自己与下级的 BP 全貌
- 目前不需要编辑功能，只读查看
- 用"BP管理"这个触发词

是这个意思吗？

用户：对。
```

### 3.3 需求澄清完成的标准

**必须明确**：
- 用户画像（谁用）
- 核心场景（在什么情况下用）
- 触发词（用户说什么词触发）
- 核心功能（至少 1-3 个）
- 不做什么（边界）

**输出**：口头确认，用户回复"对"或"是的"。

## 5. S3: 方案设计（Design）

### 5.1 设计总工的任务

设计总工负责：
1. **技术方案**：如何实现（架构、技术栈）
2. **数据流**：数据从哪里来，到哪里去
3. **接口规范**：涉及哪些 API，如何调用
4. **文件结构**：目录结构、文件组织
5. **用户体验**：失败兜底、错误处理

### 5.2 唯一真相源验证（强制）

**如果 Skill 涉及外部 API**，设计总工必须：

1. **用 curl 获取官方文档**：
   ```bash
   curl -sL "https://github.com/xgjk/dev-guide/raw/main/..."
   ```

2. **验证 API 规范**：
   - 接口路径、方法、参数、响应格式
   - 与代码实现（如果已有）是否一致

3. **输出 DESIGN-01** 时，必须包含：
   - 官方文档链接
   - 关键 API 摘要
   - 代码实现与官方文档的一致性声明

**反面案例**：
- 凭记忆推断 API 规范
- 使用本地 `api-endpoints.md` 作为权威来源
- 使用 web_search/web_fetch 获取文档

### 5.3 输出 DESIGN-01

```markdown
# DESIGN-01 - [Skill名称] 设计文档

## 技术方案
- 技术栈：[Python/Node.js/其他]
- 架构：[架构图或描述]
- 核心依赖：[依赖列表]

## 数据流
- 数据来源：[API/文件/数据库]
- 数据处理：[处理逻辑]
- 数据输出：[输出格式]

## API 规范
- 涉及的 API 列表
- 官方文档链接
- 关键接口摘要

## 文件结构
```
skill-name/
├── SKILL.md
├── scripts/
│   ├── main.py
│   └── utils.py
├── references/
│   └── ...
└── design/
    ├── DESIGN.md
    └── DISCUSSION-LOG.md
```

## 用户体验
- 正常流程：[步骤]
- 异常处理：[失败兜底]
- 错误提示：[用户友好的错误信息]

## 质量标准
- 响应时间：[要求]
- 准确性：[要求]
- 可用性：[要求]
```

### 5.4 用户确认

**必须**：
1. 向用户展示 DESIGN-01 摘要
2. 请求确认："设计方案是这样，确认后我开始开发。"
3. 用户回复"确认"或"是的"后，进入 S4

## 6. S4: 开发实现（Development）

### 6.1 交付总管的任务

交付总管负责：
1. **创建文件结构**：按 DESIGN-01 创建目录和文件
2. **编写核心代码**：实现核心功能
3. **编写 SKILL.md**：遵循三层披露原则
4. **编写测试用例**：正常、边界、异常场景

### 6.2 SKILL.md 编写规范

**三层披露原则**：

| Tier | 内容 | 加载时机 | 预算 |
|------|------|----------|------|
| 1 | frontmatter description | 每次对话都加载 | ~100 词 |
| 2 | SKILL.md 正文 | Skill 触发后加载 | 80 行以内 |
| 3 | references/ 辅助文件 | 按需加载 | 不限 |

**SKILL.md 必须包含**：
```yaml
---
name: skill-name
description: [3-5 个具体触发词，覆盖口语化表达]
homepage: https://github.com/evan-zhang/agent-factory
issues: https://github.com/evan-zhang/agent-factory/issues/new?labels=skill-name
version: "1.0.0"
tools_provided:
  - name: tool1
    description: [工具描述]
    status: active
metadata:
  requires:
    env:
      - ENV_VAR
---
```

**涉及外部 API 的 Skill，必须包含**：
```markdown
## API 文档

> ⚠️ **重要提示**：本 Skill 调用玄关开放平台 API，权威文档是玄关开放平台官方文档。
>
> **官方文档链接**：https://github.com/xgjk/dev-guide/...
>
> 本 Skill 的代码实现与官方文档完全一致。如需查看完整的 API 规范，请直接访问上述官方文档。
```

### 6.3 代码实现规范

**必须遵循**：
1. **唯一真相源原则**：代码实现与官方文档完全一致
2. **错误处理**：所有 API 调用都有 try-except
3. **日志输出**：关键步骤有日志输出
4. **环境变量**：敏感信息通过环境变量传递
5. **类型提示**：Python 代码使用类型提示

### 6.4 输出 CODE-01

```markdown
# CODE-01 - [Skill名称] 代码实现总结

## 文件清单
- SKILL.md：[行数]
- scripts/main.py：[行数]
- scripts/utils.py：[行数]
- references/：[文件列表]

## 核心功能实现
- 功能1：[实现方式]
- 功能2：[实现方式]
- 功能3：[实现方式]

## 技术债务
- [技术债务1]
- [技术债务2]

## 测试用例
- 正常场景：[描述]
- 边界场景：[描述]
- 异常场景：[描述]
```

### 6.5 用户确认

**必须**：
1. 向用户展示 CODE-01 摘要
2. 请求确认："代码开发完成，确认后我开始测试。"
3. 用户回复"确认"或"是的"后，进入 S5

## 7. S5: 质量验证（Testing）

### 7.1 质检总监的任务

质检总监负责：
1. **功能测试**：正常场景、边界场景、异常场景
2. **质量审计**：10 维度 Audit Checklist
3. **安全检查**：凭证、敏感信息、扫描触发词
4. **文档检查**：SKILL.md、references/ 完整性

### 7.2 10 维度 Audit Checklist

**必须全部通过才能进入 S6**：

### 维度 1 — 触发质量
- [ ] description 说清楚了做什么、什么时候用？
- [ ] 含 3-5 个具体触发词，覆盖口语化表达？
- [ ] 用动词开头，不以 "Use when..." 开头？

### 维度 2 — 是否真的需要建 Skill
- [ ] 这是可复用的需求，不是一次性任务？
- [ ] 没有现成工具或 Skill 可以直接覆盖？
- [ ] 不是 subagent 任务？

### 维度 3 — SKILL.md 纪律
- [ ] SKILL.md ≤ 80 行，容易扫描？
- [ ] 只包含核心工作流和规则，细节已推入 references/？

### 维度 4 — 写法质量
- [ ] 命令式语气，不是"可以考虑"？
- [ ] 关键步骤解释了"为什么"？

### 维度 5 — 最小化原则
- [ ] 目录是满足需求的最小结构？
- [ ] references/ 文件 ≤ 3 个？

### 维度 6 — 文件卫生
- [ ] 无 README/CHANGELOG 等对 AI 无用的人类文档？
- [ ] 无废弃的占位文件或实验残留？

### 维度 7 — 规范漂移检查
- [ ] 引用的文件路径真实存在？
- [ ] 示例代码/命令仍然准确有效？

### 维度 8 — Smoke Test（三场景必过）
- [ ] 正常触发：标准场景，触发且行为正确？
- [ ] 边界触发：边缘场景，触发且能处理？
- [ ] 不应触发：相似但不属于该 Skill 的请求，不触发？

### 维度 9 — Do-nothing Check
- [ ] 改一个文件能解决的，没有错误地建 Skill？

### 维度 10 — 安全与元数据（工厂专项）
- [ ] 无 process.env 写入或读取凭证？
- [ ] 涉及外部 API 的 Skill，是否指向官方文档链接？
- [ ] frontmatter metadata.requires.env 已声明必需变量？
- [ ] references/maintenance.md 存在且包含 GitHub 地址？

### 7.3 唯一真相源验证（强制）

**如果 Skill 涉及外部 API**，质检总监必须：

1. **用 curl 获取最新官方文档**：
   ```bash
   curl -sL "https://github.com/xgjk/dev-guide/raw/main/..."
   ```

2. **对比代码实现**：
   - 接口路径、方法、参数、响应格式是否一致
   - 如有差异，以官方文档为准，拒绝通过

3. **检查 SKILL.md**：
   - 是否指向官方文档链接
   - 是否有本地 API 端点表（如有，要求删除）

### 7.4 输出 TEST-01

```markdown
# TEST-01 - [Skill名称] 测试报告

## 测试环境
- 环境：[环境信息]
- 测试时间：[时间]

## 功能测试
- 正常场景：[结果]
- 边界场景：[结果]
- 异常场景：[结果]

## 质量审计
- Audit Checklist 结果：[PASS/REVISE]
- 问题清单：[问题列表]

## 安全检查
- 凭证检查：[结果]
- 敏感信息检查：[结果]
- 扫描触发词检查：[结果]

## 唯一真相源验证
- 官方文档链接：[链接]
- 代码一致性：[一致/不一致]
- SKILL.md 文档：[合规/不合规]

## 结论
- [ ] 通过

```

### 7.5 用户确认

**必须**：
1. 向用户展示 TEST-01 摘要
2. 请求确认："测试通过，确认后我开始发布。"
3. 用户回复"确认"或"是的"后，进入 S6

## 8. S6: 四重发布（Release）

### 8.1 交付总管的任务

交付总管负责四重分发：
1. **物理发布**：迁移到 `05_products/`
2. **云端发布**：ClawHub 注册
3. **企业发布**：内部 Skill 市场
4. **业务交付**：推送样件给用户

### 8.2 物理发布（Local On-boarding）

**动作**：将开发成果从 `04_workshop` 迁移到 `05_products/`

```bash
# 假设 Skill 在 04_workshop/skill-name/
cp -r 04_workshop/skill-name/ 05_products/skill-name/
```

### 8.3 云端发布（ClawHub Registry）

**动作**：执行 `clawhub publish`

```bash
clawhub publish 05_products/skill-name --changelog "v1.0.0 初始版本：[功能描述]"
```

**验证**：
- 访问 ClawHub 验证发布成功
- 检查版本号、描述是否正确

### 8.4 企业发布（Internal Skill Market）

**动作**：使用 `create-xgjk-skill` 工具链

```bash
python3 create-xgjk-skill/scripts/skill-management/publish_skill.py --skill-path 05_products/skill-name
```

**输出**：必须保留
- Skill ID
- Skill code
- downloadUrl
- isInternal

**引导用户**：
- 发布成功后，引导 Evan 访问 `https://skills.mediportal.com.cn/` 核验

### 8.5 业务交付（Artifact Delivery）

**动作**：推送"工业级业务样件"给用户

**样件内容**：
- Skill 使用示例
- 测试报告摘要
- 链接到官方文档

### 8.6 输出 RELEASE-01

```markdown
# RELEASE-01 - [Skill名称] 发布记录

## 版本信息
- 版本：v1.0.0
- 发布时间：[时间]
- 发布人：[交付总管]

- 物理发布：✅ 已迁移到 05_products/
- 云端发布：✅ ClawHub 已注册
- 企业发布：✅ 内部 Skill 市场
  - Skill ID: [ID]
  - downloadUrl: [URL]
- 业务交付：✅ 已推送样件给用户

## 变更日志
- v1.0.0：[变更描述]

## 安装/更新命令
```bash
clawhub install skill-name

```

## 反馈渠道

### 8.7 用户确认

**必须**：
1. 向用户展示 RELEASE-01
2. 请求确认："发布完成，Skill 已就绪。"
3. 用户回复"确认"或"是的"后，进入 S7

## 9. S7: 交付确认（Acceptance）

### 9.1 工厂调度员的任务

工厂调度员负责：
1. **确认用户验收**：用户是否满意
2. **处理反馈**：如有问题，回到对应阶段修复
3. **标记完成**：项目正式交付

### 9.2 用户验收标准

**用户需要确认**：
- [ ] Skill 功能符合需求
- [ ] 使用体验良好
- [ ] 文档清晰易懂
- [ ] 可以开始使用

### 9.3 反馈处理

**如有反馈**：
1. 记录反馈到 `DISCUSSION-LOG.md`
2. 判断反馈类型（需求变更、Bug、体验问题）
3. 返回对应阶段修复
4. 重新测试和发布

**无反馈或反馈已解决**：
1. 标记项目完成
2. 进入 S8

## 11. 关键原则回顾

### 11.1 唯一真相源原则（Source of Truth）

**涉及外部 API 的 Skill，必须以官方文档为唯一权威来源。**

**强制要求**：
1. **API 规范必须来自官方文档**：优先用 curl 直接获取官方 GitHub 文档
2. **禁止使用不可靠的文档来源**：本地文档、web_search、记忆推断
3. **SKILL.md 文档处理**：不维护本地 API 端点表，指向官方文档链接
4. **代码验证**：代码实现必须与官方文档完全一致

### 11.2 三层披露原则（Token 经济学）

细节只能向下移，不能向上堆。

| Tier | 内容 | 加载时机 | 预算 |
|------|------|----------|------|
| 1 | frontmatter description | 每次对话都加载 | ~100 词 |
| 2 | SKILL.md 正文 | Skill 触发后加载 | 80 行以内 |
| 3 | references/ 辅助文件 | 按需加载 | 不限 |

### 11.3 用户确认原则

每个阶段（S2-S7）完成后，必须：
1. 向用户展示阶段产出摘要
2. 请求确认："这是[阶段名称]的结果，确认后我继续。"
3. 等待用户回复"确认"或"是的"后，才进入下一阶段

### 11.4 质量门控原则

S5 测试阶段必须通过 10 维度 Audit Checklist 才能进入 S6 发布阶段。
- 任何一个维度不通过，返回 S4 修改
- 修改后重新测试
- 全部通过后才能发布
