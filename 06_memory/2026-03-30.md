## 一、经验教训：玄关系 API 调用的 URL 完整性校验（P0）

**背景**：连续两次在调用 BP/CWork API 时因 URL 路径不完整导致调试浪费。

**错误模式**：
1. 第一次：直接调用 `/getSimpleTree`（缺少 `/bp/task/v2/` 前缀）→ 报 401
2. 第二次：直接调用 `/getGoalAndKeyResult`（缺少 `/bp/task/v2/` 前缀）→ 报 401

**根因**：
- 看到第三方代码（三省交付物）里的 API 路径，直接信任并使用
- 没有先查 **dev-guide 仓库官方文档** 确认完整路径
- 没有意识到玄关系 API 是分模块的（`/bp/`、`/bp/task/v2/`、`/work-report/` 等）

**正确流程（必须遵守）**：
1. **第一步**：curl 访问 dev-guide 仓库获取官方 API 文档
   - 仓库：`https://github.com/xgjk/dev-guide`
   - BP API：`02.产品业务AI文档/BP/BP系统API说明.md`
   - CWork API：`02.产品业务AI文档/工作协同/工作协同API说明.md`
2. **第二步**：从官方文档确认**完整 URL 路径**（含所有前缀）
3. **第三步**：调用前打印完整 URL，目视检查
4. **第四步**：如有第三方代码，对比官方文档标记差异

**禁止行为**：
- ❌ 直接使用第三方代码中的 API 路径，不查文档
- ❌ 凭记忆或猜测构造 API 路径
- ❌ 看到 401 就认为是权限问题，不检查路径

**已记入 RULES.md 第十五章**。

### 核心发现

BP 三层结构是**递归的**：
- 上一层级的"举措" = 下一层级的"目标"
- 层级之间通过"举措→目标"的映射递归连接

```
BP（目标）
  └── KR（成果）
       └── 举措 = 下一层级的"目标"
              └── KR（下一层级的成果）
                   └── ...
```

### 待确认方案

建议按递归深度分配权重（越深权重越高）：
```
深度1：20%
深度2：25%
深度3：30%
深度4+：25%
```
