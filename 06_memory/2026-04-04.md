## 康哲规则要点（BP 系统）

### 拆解层级
- 集团层：拆解到 2 层（目标 → 关键成果）
- 中心/部门/员工层：拆解到 3 层（目标 → 关键成果 → 关键举措）

### 目标（O）规则
- 只写结果状态，禁动词
- 符合 SMART 原则
- 与上级目标对齐

### 关键成果（KR）规则
- 必须符合 SMART
- 配置衡量标准
- 可量化、可追踪

### 关键举措（Task）规则
- 必须可指派、可承接、可追踪
- 明确责任人和完成时间
- 与上级 KR 强关联

## Skill 创作规范（必须遵循）

### 新规范：必须声明 tools_provided

如果 Skill 有 `scripts/` 目录（即有可执行脚本），必须在 `SKILL.md` 的 YAML frontmatter 里添加 `tools_provided` 声明。

**格式**：
```yaml
---
name: my-skill
description: "这个 skill 是干嘛用的，When to use 时自动加载"
tools_provided:
  - name: my_tool
    category: exec
    risk_level: medium
    permission: exec
    description: "这个工具是做什么的"
    status: active
---
```

**字段说明**：
| 字段 | 必填 | 说明 |
| ---- | ---- | ---- |
| name | 是 | 工具唯一名称 |
| category | 是 | 类别：search/file/exec/memory/system/network |
| risk_level | 是 | 风险：low/medium/high/critical |
| permission | 是 | 权限：read/write/exec/admin |
| description | 是 | 一句话说明工具功能 |
| status | 否 | 默认 active |

**多层架构**：
- Gateway 层：`~/.openclaw/skills/` — 所有 agent 共享
- Agent 层：`~/.openclaw/workspace/skills/` — 单个 agent 私有

**快速检查清单**：
创建/更新完 skill 后，运行：
```bash
python3 skills/skill-tool-registry/scripts/register_tools.py register <skill-name>
```
确认显示「X tools declared」且「X tools registered」。

**已注册的工具**：

- cms-cwork: `cwork_client`, `cwork_api`, `cwork-search-emp`, `cwork-send-report`, `cwork-query-report`, `cwork-create-task`, `cwork-review-report`, `cwork-query-tasks`, `cwork-nudge-report`, `cwork-todo`, `cwork-templates`

## Skill 分工

### bp-prototype
- 用途：生成 BP 模板母版（年报/半年报/季报/月报）
- 触发：/bp-prototype 或"生成BP模板"
- 产出：空白模板文件

### bp-reporting-templates
- 用途：从 BP 数据生成填写模板
- 触发：/bp-report 或"生成BP报告模板"
- 产出：带数据的填报模板

### bp-manager
- 用途：日常 BP 管理（查看、新增、检查、提醒）
- 核心场景：
  - 查看自己/下级的 BP
  - 新增关键成果/关键举措
  - AI 检查（基于康哲规则）
  - 延期提醒（手工触发）
  - 查看汇报历史
  - 搜索 BP 内容

### cms-cwork
- 用途：CWork 工作协同 API 封装
- 触发词：cwork/CWork/工作协同/发送汇报/汇报/申请/周报/待办/任务/催办/搜索员工
- 核心功能：搜索员工、发送汇报、查询汇报、创建任务、审阅汇报、查询任务、催办闭环、待办管理、模板管理
