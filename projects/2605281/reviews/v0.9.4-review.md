## 关键问题

### 1. [严重度：高] 商机 ID 校验规则未经玄关实际数据验证
**问题描述**：
- 当前校验规则：`^[A-Za-z0-9_-]{3,64}$`，拒绝中文、空格、特殊符号
- 评审材料中的示例 `CP202412200012`、`CP202601120013` 均符合此规则
- 但评审未发现任何对玄关真实商机 ID 格式的验证（如查数据库、问官方、跑历史数据采样）

**修复建议**：
- 必须（1）向玄关官方确认真实商机 ID 格式规范，或（2）采样至少 100 条历史商机 ID，跑 `grep` 统计格式分布，验证当前规则覆盖率
- 如果真实数据包含特殊字符（如下划线、点号），需扩展正则或调整策略
- 验证结果写入 `design/VALIDATION-v0.9.4.md`，作为 v0.9.5 输入

**为什么必须修复**：
- 当前规则是假设，未经实证。如果玄关真实商机 ID 含特殊字符（如 `CP.2024.12.20.0012`），会直接拒绝合法商机
- v0.9.3 三轮复盘教训："产出物存在性校验" —— 这里是"外部约束存在性校验"

---

### 2. [严重度：中] YYYYMM 路径语义可能引发用户混淆
**问题描述**：
- 路径模板：`{ROOT}/{YYYYMM}/{caseCode}/`，示例 `CPYJ/202606/CP202412200012/`
- 评审材料明确：**YYYYMM 不是商机 ID 内的业务日期，是系统当天时间（项目被正式评估的月份）**
- 用户场景：12 月 31 日跑一个 1 月的商机 ID `CP202401010001`，实际归档路径 `CPYJ/202612/CP202401010001/`
- 潜在问题：用户期望按"商机月份"归档，但实际按"评估月份"归档，两者可能相差 11 个月

**修复建议**：
- **方案 A（推荐）**：从商机 ID 解析月份，如 `CP202412200012` → `202412`，路径 `CPYJ/202412/CP202412200012/`；兜底时用系统当天
- **方案 B**：在 SKILL.md / config.yaml 显式标注"YYYYMM = 评估月份，非商机月份"，并在用户文档中强调
- **方案 C**：新增 `--business-date` 参数，允许用户显式指定业务月份，优先级高于系统当天

**为什么必须修复**：
- 知识库归档是用户长期交互的核心路径，语义模糊会导致后续检索困难
- 评审材料已识别此风险，但未提供缓解方案，违反"风险必须闭环"原则

---

### 3. [严重度：中] 测试脚本变量泄漏（脚本语法检查通过，但有逻辑隐患）
**问题描述**：
- `scripts/test-run-opportunity.sh` 测试 18 后输出 `scripts/test-run-opportunity.sh: line 215: OUT4: unbound variable`
- 虽然测试用例本身标记为通过（"18/18"），但脚本存在变量未定义后使用
- 这是脚本 `set -euo pipefail` 严格模式下的真实隐患

**修复建议**：
- 检查 `test-run-opportunity.sh` 第 215 行附近的变量声明和使用，确保所有变量在使用前初始化
- 建议在测试套件中增加"脚本语法检查 + 变量泄漏检查"用例
- 修复后重新跑 `bash scripts/test-run-opportunity.sh`，确认无 unbound variable 警告

**为什么必须修复**：
- v0.9.3 教训："测试覆盖不等于质量门控" —— 18/18 通过不代表无隐患
- 测试脚本是质量门控的守门员，自身有问题会掩盖真实问题

## 重要发现（非阻塞性）

### 4. [严重度：低] 兜底格式 YYMMDD-HHMMSS 秒级粒度可能冲突
**问题描述**：
- 兜底格式 `YYMMDD-HHMMSS` 在同一秒内重复调用会自动加 `-1 / -2` 后缀
- 评审材料提到：高并发场景（如 cron 同一秒触发多次）所有 case 会带后缀，主流程 caseCode 不纯净

**当前处理**：
- 测试用例 17 已验证冲突后缀生效（`TESTDUP-$$-1` → `TESTDUP-$$-1-1`）
- 评审材料认为这是"设计上可接受的边界情况"

**建议**：
- 文档中显式说明：未传 `--opportunity` 时，caseCode 可能含后缀（如 `260614-175645-1`），业务程序不应假设格式纯净
- 高并发场景建议业务程序显式传 `--opportunity` 或加锁
