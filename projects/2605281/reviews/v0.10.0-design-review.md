**关键问题**（最多 5 个）

1. **[严重度：高] search.sh 300 行无法覆盖所有业务逻辑** → 修复建议：拆分架构，`search.sh` 只做调用编排，T1-T3 分级、关键词模板、字段抽取、评分逻辑各拆独立模块（~5-7 个小脚本），每个 50-100 行。300 行单一脚本会变成维护噩梦。

2. **[严重度：中] 字段抽取自动化可行性未验证** → 修复建议：方案声称"结构化字段抽取"是业务特定能力，但未验证 LLM 在 FDA/EMA 官方文档上抽取"批准日期/适应症/持有人"的实际成功率。**必须先做技术验证（抓 10 篇 FDA 标注文档测准确率）**，而不是假设可行。准确率 <80% 的字段必须标记为"人工验证必填"。

3. **[严重度：中] 降级链裁剪判断错误** → 修复建议：multi-search 的 3 个 provider yaml（minimax/open_websearch/web_fetch）**不是冗余配置**，而是不同引擎的参数映射和配额隔离层。内化版直接调 OpenClaw 内置 web_search 会失去配额控制、超时隔离、参数映射的灵活性。必须重新评估：是真的"裁剪"还是"移除必要抽象"？

4. **[严重度：低] 22 个 skill 改 22 处的实际成本低估** → 修复建议：方案称"2h 自动化脚本批量加"，但未考虑：（1）22 个 skill 的搜索触发词（site:nmpa.gov.cn / FDA / EMA）分布**不均匀**（实测发现：A-1 有 35 处 FDA/EMA，A-2 有大量国谈/医保，其他 skill 可能极少），（2）批量加脚本必须做差异分析，不能"22 × 20 行"统一改。**必须先跑统计分析再设计批量逻辑**。

5. **[严重度：低] orchestrator 校验插入点未明确** → 修复建议：方案称"orchestrator-resume.sh 加 Gate 校验"，但未明确在**哪个函数/哪一行**插入。实测 orchestrator-resume.sh 是 207 行智能续跑器，核心逻辑在 `find_resume_point()` 和 `mark_gate()`。必须在方案中明确：校验是在 `start-phase.sh` 调用前插入，还是在 `mark_gate()` 成功后插入？未明确 = 实现歧义。

## 评审者立场

我评审的是**设计方案**，不是 multi-search 本身。multi-search 的 3 个 provider yaml 是否冗余，取决于"配额控制/参数映射"是否为 bd-eval-cms 的业务需求。

如果 bd-eval-cms 的业务需求是"简化调用、忽略配额"，那么裁剪是合理的。但如果 bd-eval-cms 未来需要配额隔离（例如 MiniMax 配额耗尽时自动降级到 open_websearch），那么移除这些 yaml 会增加未来重写的成本。

**方案必须明确**：裁剪是"简化调用"的主动选择，还是"移除冗余"的误判。如果是前者，方案应加注："接受失去配额控制，换调用简化的设计权衡。"如果是后者，需重新评估。
