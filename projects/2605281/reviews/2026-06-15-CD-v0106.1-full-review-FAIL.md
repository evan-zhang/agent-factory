**关键问题**（最多 5 个）

1. [严重度：高] Phase 5.5 端到端主链路仍然跑不通：`start-phase.sh` 在 `phase-5-5-html` 分支只执行 `preflight-phase.sh` 后直接调用 `sync-to-knowledge-base.sh`，没有调用 `render_report.sh` 生成 `REPORT.html`；而 `sync-to-knowledge-base.sh` 默认 archive 模式又要求 `REPORT.html` 存在且 `state.json.gateStatus.phase-5-5-html=completed`，此时该 gate 刚被标成 `in_progress`，会自相矛盾地拒绝归档 → 修复建议：Phase 5.5 应显式串行为 `preflight --mode render → render_report.sh → 标记/或允许 archive 前检查 REPORT.html → sync`，并新增端到端回归测试覆盖该顺序。
2. [严重度：高] 知识库普通文件上传链路的系统性鉴权问题未解决：`sync_file()` 仍用 `POST /document-database/file/uploadContent` + `appKey` header；评审材料已确认返回 `100002: 访问令牌为空`，说明这不是偶发网络问题，而是接口鉴权方式与脚本不匹配 → 修复建议：确认 uploadContent 是否必须使用 `access-token`，若是则先 `user/login/appkey` 换 token 后统一带 `access-token`；在未确认前不要宣称“全量同步可用”。
3. [严重度：中] SOP 与脚本严重偏离，AI 照 SOP 操作大概率失败或走错路径：SOP 写 `yq` 读取配置、`$DOCVIEWER_KB_APPKEY` 环境变量、手工 curl 上传和 `{YYYYMM}/{caseCode}` 路径；脚本实际用 python3 解析配置、读取 `.secrets/kb_appkey`、使用 `CPYJ/{YYYYMM}/{caseCode}`，且 config.yaml 底部注释仍说读环境变量 → 修复建议：以脚本为准重写 SOP Phase 5.5，仅保留一条推荐命令，并同步修改 config 注释。
4. [严重度：中] `saveFileByPath` 返回 `data=null` 时用 `placeholder_${RESOURCE_ID}` 作为 `FILE_ID` 的容错不可靠：后续 `doc-preview/api/preview/ticket` 的 `bizId` 很可能要求真实 KB fileId，placeholder 不是服务端实体 ID，可能必然获取不到 previewUrl；当前逻辑会把绑定“可能成功”与 preview 能用混在一起 → 修复建议：data=null 时应通过 list/search/query 节点接口反查真实 fileId，或明确降级为“已上传但无预览链接”，不要用 placeholder 调 preview。
5. [严重度：中] KB 路径构建相对收敛但仍未完全防错：`run-opportunity.sh --opportunity` 会拒绝空白和路径分隔符，但 `sync-to-knowledge-base.sh` 第二参数未做同等 sanitize；同时 SOP 允许人工传 `{案件代号}`，AI 仍可传入错误月份/错误 caseCode，导致上传到错误目录 → 修复建议：sync 内部也复用 caseCode 校验，优先只允许从 `state.json.caseCode`/入口生成值读取；手动覆盖需显式 `--case-code` 并拒绝 `/`、空白、引号等字符。

**复审结论补充**

- P0 管道子 Shell bug：已用 here-string `done <<< "$(echo "$MANIFEST_ITEMS")"` 修复，主 Shell 能接收 `FAIL_ITEMS/PASS_COUNT` 更新，方向正确。
- preflight 循环依赖：`preflight-phase.sh` 调用 `verify-manifest.sh "$CASE_DIR" --mode render`，manifest 中 render gates 不含 `phase-5-5-html`，该点已修复。
- `validate_gate_search` 恢复：已在 preflight 后恢复调用，且按当前实现仅警告不阻断渲染；与“manifest 暂不覆盖 evidence”的过渡策略一致。
- JSON 输出注入：仍存在，属于本地工具中等风险；若 `--json` 用于自动化消费，仍建议尽快改为 argv/env 传参或纯 Python 实现。
- sync 无参数：已改为 `CASE_DIR="${1:-}"` 并输出用法，友好报错到位。

**最重要的一条建议**

先把 Phase 5.5 收敛成唯一可执行链路：`preflight(render) → render_report.sh → sync(access-token 上传 + 真实 fileId preview) → state.json 回写`，并让 SOP、config 注释、入口脚本全部只指向这一条链路。
