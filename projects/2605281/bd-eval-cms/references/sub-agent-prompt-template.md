# CMS 投前评估 - 子Agent Prompt 模板

> **用途**:Phase 3 逐 Gate 深度评估时,作为 sessions_spawn 的 task 参数基础。
> **版本**:v0.1

---

## 模板使用说明

本模板为 CMS 投前评估体系的 Gate 章节撰写子Agent 提供统一的 prompt 结构。

**使用方式**:
1. 复制下方模板
2. 替换 `{变量}` 部分为实际值
3. 通过 sessions_spawn 发送给子Agent

---

```

## 已有资料

请先读取以下文件获取上下文:

7. 本 Gate 上一版文件(增量更新时):{品种目录}/02-gate-by-chapter/history/{Gate文件名}.v{N}.md

## 本 Gate 评估要点

{从技能定义文件中提取的本Gate评估要点,列出核心评估维度}

## 撰写规范

### 1. TL;DR 5 字段（必须，放在章节顶部，取代旧版"结论卡"）

> **v0.7.1 新增**：本字段取代 v0.7.0 旧版"结论卡"格式（不共存）。
> `merge-report.sh` 会校验 5 字段，缺一报警并退出（exit 2）。
> 详细约束见 SOP.md §6.5.2。

```markdown
# G{N} {章名}

## TL;DR

| 字段 | 值 |
|------|----|
| **评级** | 推进 / 条件推进 / 停止 |
| **评分** | 0-100 分（按技能定义）|
| **关键风险** | 1-3 条最关键风险（含 C/D 级数据披露）|
| **推荐路径** | 推进/调整/终止 + 关键条件 |
| **下一步** | 谁、什么时间、做什么 |

---

## 详细评估

### {N}.1 {子节}
...
```

**重要**：TL;DR 表格 5 个 key 必须全部存在，缺任一 → merge 报错。

### 2. 结论卡（兼容旧版，已被 TL;DR 取代）

> **v0.7.1 起**：原"结论卡"格式（blockquote 语法）由 TL;DR 5 字段取代。
> 旧版结论卡仍可保留作为补充说明，但不再是必需首部元素。
> 完整标记语法约定仍见 SKILL.md Step 10（仅供历史参考）。

```
> **【Gate {N}:{名称}】**
> 结论:✅通过 / ⚠条件通过 / ❌停止 / ⏳待验证
> 置信度:高 / 中 / 低
> 关键支撑证据:
>   1. ...
>   2. ...
>   3. ...
> 需补充证据 Top 5:
>   1. ...(预计获取难度:高/中/低)
>   2. ...
> 红旗事项:(如有，用 🚩 标记)
> 下一步行动:
```

### 3. 置信度标注

对关键数据(市场规模、患者人数、竞品销售额、峰值销售预测等):

| 等级 | 定义 | 标注方式 |
|------|------|---------|
| A级 | ≥2个独立来源交叉验证 | 无需标注 |
| B级 | 单一权威来源 | 标注来源名称,如(来源:IQVIA 2025)|
| C级 | 仅内部/合作方提供 | `[C级-待验证]` |
| D级 | 基于假设推算 | `[D级-基于假设]` |

**关键财务数据使用 C/D 级时**,必须在章节末尾添加风险提示段。

### 4. 信息来源标注

每段实质性内容必须标注来源:

- 正文内引用标注 `[{前缀}-XXX]`（如 `[G2-003]`、`[P1-001]`、`[EXT-002]`），指向对应前缀的目录下的文件
- `来源：内部评估——[文件名]（日期）`（非公开来源时使用）
- `需进一步与[对象]沟通/确认`
- **禁止**：不允许出现"外网核查""分析推断"等无具体引用编号的标注

**必须进行外网独立核查**，至少覆盖：权威指南、竞品注册状态、合作方合规历史、医保/准入政策、市场数据交叉验证。所有核查结果必须存入对应前缀的 references 文件并在正文中引用。

### 5. 阶段标签

- 基于公开信息的内容标注 `[阶段A]`
- 基于深化数据的内容标注 `[阶段B]`
- 同一章节内两种内容都有时,分别标注

### 6. 财务缩写规范

首次出现必须标注中文全称:
- IRR(内部收益率)
- NPV(净现值)
- COGS(销售商品成本)
- rNPV(风险调整净现值)
- WACC(加权平均资本成本)
- EBITDA(税息折旧摊销前利润)
- VBP(带量采购)

### 7. 信息冲突处理

当内部评估与外网核查存在冲突时:
1. 两者并列呈现
2. 给出独立判断及依据
3. 以外网核查为基准
4. 添加冲突标记

### 8. 禁止事项

- 禁止灰色字体
- 禁止元描述("本章节旨在...")
- 禁止修订说明
- 禁止未标注来源的市场数据

## 外部资料优先规则

如果 `{品种目录}/references/EXT/` 目录存在且非空:
1. 先列出 EXT/ 目录下已有文件,识别覆盖本 Gate 评估维度的外部资料
2. 对已覆盖维度:联网搜索减少为 1-2 次交叉验证
3. 对未覆盖维度:正常执行 3+ 次定向搜索
4. 引用外部资料时使用 [EXT-XXX] 编号
5. 在章节开头列出外部资料来源清单

如果本 Gate 是增量更新(history/ 下有旧版本文件):
1. 先读取旧版本,理解上一版的内容和结论
2. 新版本中明确标注与上一版的差异(新增/修正/删除的数据点)
3. 如果旧版结论与新数据矛盾,必须在结论卡中说明变更原因

## 搜索要求 + 参考文献采集

- 至少进行 3 次定向搜索(覆盖本 Gate 核心评估维度)
- **搜索前**:先列出 `{品种目录}/references/{前缀}/` 目录下已有文件,了解已抓取 URL 避免重复,以及引用前面节点的数据时使用对应前缀编号(如 `[P1-001]`、`[G3-005]`)
- **搜索后**:对每个有价值结果(含具体数据/结论)执行 `web_fetch` 抓取内容
- 新的参考文献写入 `{品种目录}/references/{前缀}/{前缀}-XXX.md`,编号从 `{前缀}-001` 开始独立递增
- 参考文献格式:
  ```markdown
  # [{前缀}-001] 文章标题
  - **URL**: https://...
  - **抓取时间**: 2026-05-29 11:00 CST
  - **来源类型**: 官方|学术|行业媒体|公司公告|监管数据库

  ## 原文内容
  (web_fetch 抓取的完整内容,尽量完整保留)

  ## 关键数据点
  - 具体数字/结论1
  - 具体数字/结论2
  ```
- **正文引用**:每个数据点后标注 `[{前缀}-XXX]`,示例:全球PBC患者约50万人[P1-012]、竞品销售额12亿美元[G3-005]
- **禁止**:不允许出现"外网核查""分析推断"等无具体引用编号的标注
- **报告开头**:列出本 Gate 使用的参考文献编号清单,如:`本 Gate 引用:[P1-003] [G1-007] [G3-012]`
- 搜索来源记录在章节末尾
- 优先使用权威来源(NMPA/FDA/ClinicalTrials/IQVIA/学术文献)

## 输出

将完整章节内容写入:{输出文件路径}

文件命名格式:`Gate-{N}-{Gate英文名}.md`
示例:`Gate-1-premise.md`、`Gate-4-payment.md`

## 约束

- 全文简体中文
- 财务缩写首次标注全称
- 关键数据置信度标注完整
- 不超过 3000 字(不含参考文献)
```

你是 CMS 投前评估体系的交叉验证专家。

## 任务

## 待验证章节

## 验证检查项

1. **结论卡格式**:每个 Gate 是否有标准结论卡
2. **Gate 间一致性**:不同 Gate 引用的同一数据是否一致
3. **置信度合理性**:标注等级是否与数据来源匹配
4. **参考文献引用**:每个数据点是否有 `[{前缀}-XXX]` 引用,是否有"外网核查"等无编号标注
5. **交叉引用**:Gate 6 是否正确引用了 Gate 1-5 的结论(使用对应前缀编号如 `[P1-001]`、`[G3-005]`)
6. **财务数据一致性**:不同章节的同一财务指标是否一致
7. **冲突标记**:内外部信息不一致处是否标记
8. **参考文献完整性**:正文引用的编号是否在对应的 `{品种目录}/references/{前缀}/` 目录下存在对应文件
9. **版本一致性**(增量更新时):更新后的 Gate 文件头部版本号是否正确递增
10. **外部资料引用**:EXT-XXX 编号是否在 `{品种目录}/references/EXT/` 目录下存在对应文件

## 输出

列出发现的问题,按严重程度排序:
- 🔴 必须修正(数据矛盾/结论冲突)
- 🟡 建议修正(置信度不合理/来源缺失)
- 🟢 可选优化(表述优化/格式调整)

写入:{输出文件路径}
```

## 全流程子Agent 通用约束（v2026.5.31）

以下约束适用于所有子Agent（全流程执行器、Gate撰写器、补跑执行器）：

### 1. 强制 state.json 更新

每完成一个 Phase，**必须立即**更新 `{品种目录}/state.json` 的 phase 字段：

```
Phase 1 完成 → phase = "discovery_complete"
Phase 2 完成 → phase = "routing_complete"  
Phase 3 完成 → phase = "evaluation_complete"
Phase 4 完成 → phase = "battle_complete"
Phase 5 完成 → phase = "report_finalized"
```

**这是最高优先级操作**。不要等到所有 Phase 都做完才更新，也不要忘记这一步。

### 2. Gate 文件命名（v0.7.1：标准名优先，允许少量兼容变体）

标准文件名如下：
```
02-gate-by-chapter/One-pager.md
02-gate-by-chapter/Gate-1-premise.md
02-gate-by-chapter/Gate-2-positioning.md
02-gate-by-chapter/Gate-3-evidence.md
02-gate-by-chapter/Gate-4-payment.md
02-gate-by-chapter/Gate-5-cost.md
02-gate-by-chapter/Gate-6-feasibility.md
```

允许的兼容变体（仅为避免旧 prompt/旧案例断裂）：
- Gate 1：`Gate-1-premise.md` / `Gate-1-precondition.md`
- Gate 6：`Gate-6-feasibility.md` / `Gate-6-dealability.md` / `Gate-6-doability.md` / `Gate-6-deal.md`

不建议新增其他随意命名；如需新增，必须先同步 SOP.md §6.5 与 `scripts/merge-report.sh`。

### 3. 执行日志

在 `{品种目录}/execution-log.md` 中记录每个 Phase 的：开始时间、结束时间、耗时、模型名、Token 用量、工具调用次数、重试次数。

### 4. Phase 5.5 HTML 验证

生成 REPORT.html 后，必须执行：
```bash
grep -c '{{' REPORT.html
```
结果必须为 0。如果不为 0，手动替换残留模板变量。

### 5. 知识库同步

HTML 验证通过后，执行知识库同步：
```bash
bash {SKILL目录}/scripts/sync-to-knowledge-base.sh "{品种目录}" "{案件代号}"
```
