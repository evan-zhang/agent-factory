## 逐节点评审

### 节点 1：SKILL.md 重写 — 评级：REVISE

**问题 1：文件体积风险被低估**

计划估算合并后 1200~1500 行。当前 SKILL.md 已有 972 行，新版 Step 1~11 内容约 470 行，加上 Phase↔Step 映射表，合并后实际可能达到 **1600~1800 行**。风险表中也提到了"SKILL.md 过大"但只给了一个模糊的"考虑拆分"缓解方案。

**建议**：在执行前明确决策——是合并为单文件（接受 1800 行），还是拆分为 `SKILL.md`（规范层，Step 1~11）+ `EXECUTION.md`（执行层，Phase 1~5.5）。如果选择单文件，需要论证 Agent 在 1800 行上下文中不会丢失关键执行指令。建议选择拆分方案，风险表中的缓解措施升级为执行方案。

**问题 2：Phase↔Step 映射表缺失设计**

计划提到"旧版 Phase 流程与新版 Step 流程产生歧义"是中风险，但没有设计具体的映射关系。这是双层架构的核心胶水层——Agent 需要知道 Phase 2 对应 Step 2+Step 8，Phase 3 对应 Step 5 等。

**建议**：在节点 1 的验证标准中增加：
- [ ] SKILL.md 包含 Phase↔Step 双向映射表（明确哪个 Phase 执行时参考哪些 Step）

**问题 3：缺少 SKILL.md frontmatter 更新**

当前 SKILL.md 的 `version: "0.1.0"`。改造后应升至 `0.2.0` 或 `1.0.0`，且 `description` 需反映双层架构。

**建议**：验证标准增加：
- [ ] frontmatter version 更新
- [ ] description 反映"规范层 + 执行层"双层架构

### 节点 4：报告模板补充 — 评级：REVISE

**问题 1：现状评估不准确**

经核查 `scripts/style-12/skeleton.html`：
- `conflict-box` — **已有**（第 156 行）
- `highlight-box` — **已有**（第 198 行）
- `gate-box` — **缺失**（0 匹配）
- `conclusion-tag` — **缺失**（0 匹配）
- `red-flag` — **缺失**（0 匹配，仅有 `veto-fail` 等替代实现）

计划假设所有 5 个样式都可能缺失，但实际上只缺 3 个。工作范围应缩小。

**问题 2：convert-md-to-html.py 的改造被低估**

当前 `convert-md-to-html.py` 只处理 `conflict-box`（第 199 行），不处理 gate-box、conclusion-tag、red-flag。这意味着不仅要补 CSS，还要补 **Markdown → HTML 的转换逻辑**——即 Agent 输出的 md 中需要约定特殊的标记语法（如 `> [GATE:1]` 或某种标记），由转换脚本将其渲染为 gate-box。

**建议**：
1. 修改验证标准，明确区分"已有"和"需新增"的样式
2. 增加子任务：定义 gate-box / conclusion-tag / red-flag 的 Markdown 标记语法约定
3. 增加子任务：在 convert-md-to-html.py 中增加对应转换规则
4. 工作量从"低"调整为"中"
