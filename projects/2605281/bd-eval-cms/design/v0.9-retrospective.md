## 1. 本轮目标

本轮原始目标不是新增一个普通样式，而是确认用户提供的 A-1 原始 HTML 模板及 TRTL-729 示例所体现的业务结构、方法学和展示要求，是否能被现有自动化报告体系稳定承接。

核对后确认：默认 style-12 不能完全满足 A-1 原始模板所代表的 CMS BD 投前报告要求，因此新增独立 style-a1，不修改 style-12 / style-13。

随后进一步确认：A-1 模板不是单一技能专用模板，而是 CMS BD 评估体系的通用报告骨架。于是本轮从“补一个 A-1 样式”升级为“建立 CMS BD 通用模板内核”。

## 4. 关键决策

### 4.1 不修改 style-12 / style-13

原因：style-12 / style-13 是既有风格资产，A-1 原始模板代表的是另一套业务报告骨架。强行改 style-12 会破坏已有行为。

决策：新增独立 style-a1。

### 4.2 A-1 模板升级为通用骨架

原因：A-1 原始模板包含 Gate 1~6、互斥规则、术语表、参考文献、信息冲突表、院内院外双轨等结构，这些不是 A-1 专属，而是 CMS BD 评估体系的共用方法学。

决策：从单模板升级为 profile-driven 通用模板内核。

### 4.3 v0.9 只做三个代表 profile

原因：工程质量优先于一次性覆盖数量。直接做 20 个 profile 容易造成低质量堆砌。

决策：v0.9 只做 A-1 / A-5 / A-7 三个代表 profile，用它们验证通用架构。

### 4.4 历史测试数据不修复

原因：历史案例数据不代表当前模板体系的真实输入契约，反复修历史数据会消耗精力且不能提升核心架构质量。

决策：优先使用 fixture + 新案例验证。

### 4.5 发布严格白名单

原因：本地存在历史脏工作区，包括 doc-viewer 删除、旧脚本改动、memory 文件等。

决策：v0.9 只提交本次必要文件，不纳入历史脏改动。

## 5. 已知不足

### 5.1 Profile 覆盖仍不完整

目前只有 A-1 / A-5 / A-7 三个代表 profile，尚未覆盖 CMS BD 全部技能。

### 5.2 输入契约还不够硬

renderer 已支持明确结构标记，但上游 Markdown 生成端尚未被 schema 化约束。

风险：规模化生产时，问题可能不出在 HTML 渲染，而出在上游 Markdown 结构漂移。

### 5.3 Gate 方法学仍偏展示层

Gate 1~6、互斥规则、置信度、财务门槛目前主要体现在报告结构中，尚未升级为规则引擎。

### 5.4 证据链检查不够强

尚未强制做到每个关键判断、每个 Gate 结论都必须绑定来源与置信度。

### 5.5 测试体系偏组件覆盖

目前测试能够证明组件渲染完整，但还没有形成真实案例 golden set、视觉回归、PDF 打印验收和失败用例矩阵。

### 5.6 工作区治理需要单独处理

本地仍有未纳入发布的历史脏改动。下一轮开发前应明确隔离或清理策略，防止误提交。

## 6. 下一版本建议

下一版本不建议直接扩展 20 个 profile。

建议定义为：

**v0.9.1：质量固化版**

核心目标：把 v0.9 的通用模板内核固化为可持续演进的工程底座。

建议范围：

1. profile schema
2. profile registry
3. 上游 Markdown 输出契约
4. renderer 失败策略
5. 真实案例验收入口
6. 工作区清理与发布边界治理

## 7. 不进入 v0.9.1 的事项

以下事项建议留到 v1.0 或更后版本：

- 20 个 CMS BD 技能 profile 全覆盖
- Gate 规则引擎
- 财务阈值配置化自动判定
- 全量证据链审计系统
- PDF / PPT 自动出版级排版
- 对历史案例数据的大规模修复

## 8. 复盘结论

v0.9 的方向是正确的：它已经把 A-1 单模板问题升级为 CMS BD 通用模板内核，并用 A-1 / A-5 / A-7 三个代表 profile 验证了 profile-driven 路径。

但 v0.9 仍只是“可验证的模板内核”，不是“年产上千份报告的生产级系统”。

下一步应按 Agent Factory AF-SOP 进入 v0.9.1 质量固化流程：先需求固化，再方案设计，再评审，再开发，不直接跳到写代码。
