# Spec Contract Delivery

将模糊需求转化为可执行、可验证、可验收的需求契约，并按 Spec-Driven Development + Contract Verification 流程指导 AI 编程 Agent 完成交付。

## 适用场景

当用户提出软件开发、功能改造、系统集成、自动化脚本、AI Agent、数据处理、前后端开发等需求时，使用本 Skill。

本 Skill 的目标不是立即写代码，而是先把需求转化为一份可执行、可验证、可验收的「需求契约」，再进入开发。

## 核心方法

> Spec-Driven Development + Contract Verification

1. 需求规格化
2. 技术方案化
3. 任务拆解化
4. 编码实现
5. 自测验证
6. 交付验收

## 工作原则

### 1. 契约优先

在需求不清楚时，不允许直接写代码。必须先输出需求澄清问题。

如果需求已经足够明确，则直接生成需求契约。

### 2. 范围受控

- 不超出需求契约范围
- 不擅自改变已有架构
- 不删除现有功能
- 不引入无关依赖
- 不做用户未要求的扩展
- 每个功能必须对应验收标准

### 3. 可验证交付

所有开发结果必须能够被验证。每个需求至少要有：

- 验收标准
- 测试场景（正常 + 异常）
- 最终交付验证报告

## 标准工作流

### Step 1：生成需求契约

输出文件：`/specs/需求契约.md`（模板见 `templates/需求契约.md`）

### Step 2：生成技术方案

输出文件：`/specs/技术方案.md`（模板见 `templates/技术方案.md`）

### Step 3：拆解开发任务

输出文件：`/tasks/开发任务.md`（模板见 `templates/开发任务.md`）

要求每个任务可独立完成、可独立验证，不允许出现"大而空"的任务。

### Step 4：编码实现

编码时必须：
1. 先读取需求契约
2. 再读取技术方案
3. 按开发任务逐项实现
4. 每次修改后列出修改文件
5. 每完成一个任务就进行局部验证

### Step 5：交付验证

输出文件：`/reports/交付验证报告.md`（模板见 `templates/交付验证报告.md`）

## 输入要求

用户可以提供以下任意一种输入：
1. 一句话需求
2. 产品需求文档
3. 页面截图
4. Bug 描述
5. 现有代码仓库
6. 接口说明
7. 会议纪要
8. 客户口头需求

Agent 必须先判断需求是否足够明确。缺失较多时必须先澄清。

## 最小可执行契约

如果用户要求快速推进，至少生成：用户、动作、系统响应、成功标准、失败情况、测试用例。

## 禁止行为

- 禁止跳过需求契约直接编码
- 禁止把不确定内容当成事实
- 禁止私自扩大功能范围
- 禁止只说"已完成"但不提供验证依据
- 禁止只写代码不写测试或验证说明
- 禁止交付时不说明风险

## 最终交付标准

一次合格交付必须同时满足：
1. 有需求契约
2. 有技术方案
3. 有任务拆解
4. 有实现记录
5. 有测试结果
6. 有交付验证报告
7. 明确说明是否可交付

## 推荐目录结构

