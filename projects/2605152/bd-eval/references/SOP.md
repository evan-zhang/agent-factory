## 设计理念

本流水线的目标：**BD品种管线初审筛选**。

**决策定位**: 这是公司投资决策流程的**第一道门槛**（初审），不是最终决断。报告的目标是回答一个问题——**这个品种值不值得投入资源做深入调研？**

后续还有多轮内部审核、与被投资方深度沟通、尽调等环节。本报告只需做到：
- ✅ 筛掉明确不靠谱的品种（一票否决触发）
- ✅ 识别值得深入的品种及其核心风险
- ✅ 为后续尽调指明重点方向

**不代表**: 不需要做最终投资决策，不需要精确到小数点的财务模型，不需要替代真实尽调。

核心原则（按优先级排序）：
1. **不放过致命风险** — 一票否决项必须严格检查，漏掉一个就是数千万沉没成本
2. **不漏掉好机会** — 有潜力的品种不应该因为搜索不充分被误判
3. **数据真实可溯源** — 所有数据有来源，不确定的必须标记
4. **对立观点必须呈现** — 正反论证都要有，让决策者看到两面
5. **效率与质量平衡** — 这是初审不是终审，报告长度够用即可，不追求面面俱到

## 完整执行流程

```
用户输入品种名称
    ↓
Phase 1: DISCOVERY（宽度搜索 + 自动模板选择）
    ↓
Phase 2: 模板选择 Battle（两个AI对抗，自动确定模板）
    ↓
Phase 3: GRV（逐章节深度搜索+撰写+交叉验证）
    ↓
Phase 4: GRV Battle（审查层 vs 执行层对抗，最多3轮）
    ↓
Phase 5: 报告合并 + 质量终检 + 推送
    ↓
用户收到完整报告（含所有争议点、风险标记、数据时效声明）
```

### Phase 1: DISCOVERY（宽度搜索 + 自动模板选择）

**执行者**: TPR Orchestrator

#### Step 0: 标的方画像确认（v5.3 新增，防 P1 事故）

**在开始搜索之前，必须向用户确认标的方信息。这是所有后续分析的基石。**

向用户询问以下信息（一次只问一个问题，附带建议和理由）：

1. **标的方是谁？** — 公司/机构名称
2. **合作模式是什么？** — 自研/License-in/代理权引进/技术平台合作/其他（建议用户提供已知信息）
3. **标的方现有产能？** — 有现有产能/计划建设/无（如已知请提供规模，如 2,000L 反应器）
4. **标的方核心优势？** — 用户已知的优势（如成本控制、渠道、注册能力等）

用户回复后，写入 `00-target-profile.md`（标的方画像）：
```markdown
# 标的方画像

## 基本信息
- 标的方名称: 
- 合作模式: 
- 现有产能: 
- 核心优势: 

## 信息来源
- 用户确认项: 
- 待验证项: 

## 对后续分析的影响
- 财务模型基准: 基于标的方实际产能/行业平均
- 竞争力评估: 基于标的方实际优势
```

**财务模型（G-6）必须基于标的方实际数据，禁止用“行业平均”替代标的方实际产能和成本结构。**

> ⚠️ 如果用户说"不知道"或"先跑吧"，允许在 `00-target-profile.md` 中标记"待确认"，但财务模型章节必须标注"⚠️ 基于行业平均，非标的方实际数据"。

#### Step 1: 创建项目目录结构

对每个品种:
```
projects/bd-eval/{药品名}/
├── state.json
├── 00-target-profile.md    ← 标的方画像（v5.3 新增）
├── 01-discovery.md
├── 02-grv-by-chapter/
├── battle/
├── 03-battle-summary.md
└── 04-final-report.md
```

#### Step 2: 宽度搜索

**宽度搜索**（web_search ≥3次），收集:
- 通用名/商品名/代号、原研公司、CAS号
- 作用机制/靶点
- 适应症
- 全球注册状态（FDA/EMA/PMDA/NMPA）
- 临床阶段和数据
- 市场规模/竞争格局概况
- 专利/知识产权概况
- 合作/授权历史

#### Step 3: 写入与模板匹配

3. 写入 `01-discovery.md`

4. **初步模板匹配**（Orchestrator根据搜索结果判断）

5. 更新 `state.json`: `phase=discovery_complete, preliminary_template=模板X`

### Phase 2: 模板选择 Battle（自动，无需用户干预）

**目标**: 防止单一AI判断失误导致模板选错。

**执行者**: TPR Orchestrator 串行调度独立子Agent

#### 流程

```
Orchestrator 给出初步模板判断 + 理由
    ↓
【模板审查层】（独立子Agent）
  - 读取 01-discovery.md
  - 读取全部7套模板的定义和适用条件
  - 判断: Orchestrator 的模板选择是否正确？
  - 如果同意 → 确认模板
  - 如果不同意 → 提出替代模板 + 理由
  输出: battle/TEMPLATE-SELECTION-AUDITOR.md
    ↓
若审查层不同意:
  Orchestrator 回应 → 审查层再确认
  若仍不一致 → 两个模板都保留，GRV 按更全面的模板执行
```

`01-discovery.md` 末尾追加模板选择说明（含选定理由、排除理由、Battle结果）。

### Phase 3: GRV（逐章节深度评估 + 交叉验证）

**执行者**: GRV子Agent（撰写） + 独立验证子Agent（交叉检查）

#### 核心约束

1. **必须逐章节处理，不得一次性写完或跳过搜索**
2. **每章至少 3 次针对性 web_search**（从v3.1的2次提升）
3. **禁止凭已有知识写章节，所有数据必须有搜索支撑**
4. **报告第一章（00-品种概述与模板说明）为必写章节**
5. **每章正文≥2,000字（v5.3 新增，防 P2 深度不足）**
6. **每章必须包含至少3个数据表格（对比矩阵、历史趋势、关键数字）（v5.3 新增）**
7. **每章必须包含至少5个具体数据点（含数字、来源、时间）（v5.3 新增）**
8. **每个核心论点必须同时呈现正向和反向两面（v5.3 新增）**

#### 报告第一章：品种概述与模板说明（必写）

每个品种报告的第一章必须是：

```markdown
# 0. 品种概述与评估框架说明

## 品种简介
（一段话描述品种是什么、谁做的、做到什么阶段了、核心卖点）

## 为什么选这个模板
（说明本品种匹配当前模板的关键特征，以及为什么排除其他模板）

## 投资视角
（一句话说明这个品种的投资类型：蓝海/FIC/快速跟随/填充/高风险高回报等）

## 投资规模与风险定性
- 预估投资总额量级（前期款+里程碑+开发成本）
- 风险定性：高风险高回报 / 中风险中回报 / 低风险低回报
- 对公司整体管线的战略价值：核心管线 / 补充管线 / 填充品种

## 初审结论预判
（基于已搜索的信息，初步判断是否值得深入调研，及后续尽调应重点关注的领域）
```

#### 逐章执行流程

```
第0章: 品种概述与模板说明（必写）
    ↓
对模板的每个章节:
  1. web_search ≥3次，搜索词针对本章节具体问题
  2. 根据搜索结果撰写本章节内容（中文）
  3. 正文关键数据标注引用编号 [1] [2]，章节末尾列参考来源（含URL）
  4. 章节末尾附「参考来源」列表（论文参考文献风格，每条含完整URL）
  5. 不确定/待验证的数据用醒目标记:
     ⚠️ [待验证] 来源未经同行评审
     🔴 [关键风险] 可能影响投资决策的重大不确定性
  6. 写入 02-grv-by-chapter/{序号}-{章节名}.md
  7. ★ 章节数据验证（独立子Agent交叉检查，见下文）
  8. 根据验证结果修正章节
  9. 进入下一章节
```

#### 引用规范（强制）

采用学术论文引用风格：正文中 `[1]` `[2]` 编号标注，章节末尾按编号列出完整参考来源（含URL）。

**正文格式** — 关键数据后标注引用编号：
```markdown
全球玫瑰痤疮市场规模约24亿美元[1]，其中美国占45%[2]。
```

每个章节末尾必须包含**参考来源列表**，按论文参考文献格式，每条附带完整URL：

```markdown
## 参考来源

[1] Grand View Research. 全球玫瑰痤疮市场报告2024. 2024.
    类型: 商业数据库 | 可信度: 🟡中 | 数据年份: 2024
    https://example.com/report-2024

[2] FDA. RHOFADE (oxymetazoline hydrochloride) Prescribing Information. 2023.
    类型: 监管文件(FDA/NMPA) | 可信度: 🟢高 | 数据年份: 2023
    https://www.accessdata.fda.gov/drugsatfda_docs/label/2023/...

[3] 盐酸羟甲唑啉乳膏中国市场分析. 2024.
    类型: 网络搜索 | 可信度: 🟠待验证 | 数据年份: 2024
    https://example.com/article-xyz
```

**规则**：
- URL 是必填项，来源就是 web_search 搜索到的那个网页链接
- 子Agent 必须原样保留 web_search 返回的 URL，不得丢弃
- 每条来源标注数据年份（读者需要知道这个数据是哪年的）

可信度标签：
- 🟢 **高**: 官方/监管文件、同行评审期刊、临床指南
- 🟡 **中**: 会议摘要、商业数据库
- 🟠 **低**: 企业公告/新闻稿、Web搜索结果

#### 数据冲突裁决规则（新增）

当多个来源对同一数据给出不同值时：
1. **优先级**: 🟢来源 > 🟡来源 > 🟠来源
2. **时效性**: 同可信度下，较新年份的数据优先
3. **处理方式**: 在正文中列出范围（如"市场规模估计50-100亿[1][2]"），并在脚注说明差异原因
4. **禁止**: 不得为了"好看"只选择对投资论点有利的数据

#### 章节数据验证（独立子Agent交叉检查）

**目标**: 防止幻觉、错误、遗漏。不是自检，是**独立AI交叉验证**。

**执行者**: Orchestrator 在GRV子Agent完成每章后，派独立验证子Agent检查。

**注意**: 为平衡质量和效率，验证子Agent可以批量处理——每完成3-5章做一次批量验证，而非每章一次。

验证子Agent任务:
```
读取 3-5 个已完成章节
对每个章节:
  1. 数据溯源检查:
     - 每个具体数字/事实是否有对应引用来源？
     - 引用来源是否真实存在？（抽查关键URL是否可访问）
     - 多个来源之间的数据是否一致？
  
  2. 幻觉检查:
     - 是否存在无法溯源的"常识性"数据？
     - 是否有AI编造的论文标题、不存在的临床试验编号？
     - 搜索结果是否被过度解读或断章取义？
  
  3. 遗漏检查:
     - 对照模板要求，本章节的评估要点是否都已涵盖？
     - 是否存在明显应该搜索但没搜索的重要信息？
  
  4. 搜索充分性检查（新增）:
     - 本章的搜索词是否覆盖了章节的核心维度？
     - 是否有维度只用了一个来源？（单源风险）
     - 建议: 如果某维度只有一个来源，要求GRV子Agent补充搜索
  
  输出: 验证报告，列出每章的:
  - ✅ 通过项
  - ⚠️ 待修正项（具体指出哪个数据、什么问题）
  - 🔴 必须修正项（幻觉/严重错误）
```

Orchestrator 根据验证报告，将修正要求发回GRV子Agent，GRV子Agent修正后重新写入。

**数据验证记录格式**（追加在每章「参考来源」之后）：
```markdown
## 数据验证记录
- 验证时间: 2026-05-15
- 验证方式: 独立子Agent交叉检查
- 幻觉检查: ✅通过 / ⚠️发现N处待验证 / 🔴发现N处错误（已修正）
- 遗漏检查: ✅通过 / ⚠️补充了以下内容: ...
- 搜索充分性: ✅每维度≥2源 / ⚠️以下维度仅单源: ...
- 未验证数据: [列出标记⚠️的具体数据和原因]
```

### Phase 4: GRV Battle（正式对抗审查）

**执行者**: 两个独立子Agent（审查层 + 执行层），由 Orchestrator 串行调度

#### Battle 认知定位

目标不是"通不通过"，而是"有没有看到盲区"。

#### 角色与流程

```
GRV全部章节完成 + 数据验证通过
    ↓
【审查层】（独立子Agent，与GRV子Agent和验证子Agent均不同）
  - 以该领域最挑剔的评审者身份审查全部GRV章节
  - 提出 3-5 个实质性异议
  - 每个异议: 引用具体章节 → 说明为什么有问题 → 提出修改建议
  - 给出裁定: APPROVE / REJECT / CONDITIONAL
  - 输出: battle/BATTLE-R1-AUDITOR.md
    ↓
若 APPROVE → 进入 Phase 5
若 CONDITIONAL → 汇总条件清单标记在报告中
若 REJECT → 进入修订循环
    ↓
【执行层】（独立子Agent）
  - 逐条回应审查层异议: 接受（说明改什么）或拒绝（附证据）
  - 更新 02-grv-by-chapter/ 中对应章节
  - 输出: battle/BATTLE-R1-EXECUTOR.md
    ↓
【审查层】重审修订版
  - 输出: battle/BATTLE-R2-AUDITOR.md
    ↓
重复最多 3 轮
    ↓
若 3 轮无法达成一致 → 将争议点在报告中用 🔴 标记
```

#### 审查层必须检查的维度（咨询级扩展）

1. **跨章节逻辑一致性**: PC-1说"风险低"，M-2说"竞品领先3年"——矛盾吗？
2. **投资论点完整性**: 正面论点和反面论点是否都充分展开？有没有只说好的不说坏的？
3. **数据验证记录复查**: 数据验证中发现的问题是否都已修正？
4. **关键假设显性化**: 报告的投资结论建立在哪些假设上？这些假设合理吗？
5. **一票否决项**: 逐一检查一票否决清单，确认每项的判定是否有充分证据
6. **商业逻辑审查**: 从药企BD团队视角审查——这个品种值得花资源做深入调研吗？
7. **估值合理性**: 财务模型的假设是否在合理区间？有没有明显偏高或偏低？
8. **后续尽调重点（新增）**: 如果决定推进，下一步应该重点查什么？哪些信息是本报告无法通过搜索获得的，必须通过与被投资方沟通获取？
9. **对标分析**: 类似品种在中国市场的引进先例和对价——我们的估值在不在合理区间？
10. **假设降权检查（v5.3 新增，防 P3）**: 财务模型各情景的支撑假设中，"待核实"项目占比是否超过50%？如果是，该情景 IRR 必须标注"⚠️ 低置信度"，不得作为主要结论依据。差异化能力的置信度必须显性评估：已确认/初步可行/待核实/不可行。

#### 审查层行为规范

1. **最挑剔的评审者身份**
2. **3-5 个实质性异议**（非格式/拼写问题）
3. **每个异议**: 引用具体章节 → 说明问题 → 提出修改建议
4. **主客观分流**: 客观错误直接标注，主观争议列入争议清单
5. **明确裁定**: APPROVE / REJECT / CONDITIONAL
6. **领域完整性审查**: 是否遗漏行业已知的重要分歧

#### 执行层行为规范

1. **逐条回应**: 接受/拒绝 + 理由
2. **接受的修改体现在章节修订中**
3. **拒绝的理由有证据支撑**

#### Battle 产出物

```

├── TEMPLATE-SELECTION-AUDITOR.md    # 模板选择审查（Phase 2）
├── TEMPLATE-SELECTION-RESPONSE.md   # 模板选择回应（如有争议）

├── BATTLE-R2-EXECUTOR.md            # 第2轮（如有）
└── BATTLE-R3-AUDITOR.md             # 第3轮（如有）
```

`03-battle-summary.md` 汇总:
- 模板选择Battle结果
- GRV Battle每轮异议+回应摘要
- 最终裁定
- **未解决争议点**（🔴 标记）
- **关键投资假设清单**（显性列出报告结论依赖的核心假设）
- **初审红绿灯**:
  ```
  🟢 值得深入 — 无致命风险，建议投入资源做尽调
  🟡 有条件推进 — 有疑虑但非致命，需在尽调中优先澄清
  🔴 不建议推进 — 触发一票否决或风险收益比明显不合理
  ```
- **后续尽调重点（新增）**: 推进后应重点查什么、哪些信息必须从被投资方获取
- **对标交易参考**: 类似品种在中国市场的引进先例和对价
- 修订清单

### Phase 5: 报告合并 + 质量终检 + 推送

**执行者**: TPR Orchestrator

#### ⚠️ Phase 5 前置条件检查（v5.3 新增，防 P0 事故）

**在启动 Phase 5 之前，必须逐条确认以下条件全部通过：**

```
Phase 5 启动条件：
✅ 所有 GRV 章节文件存在于 02-grv-by-chapter/（对照模板检查）
✅ Battle 执行层（BATTLE-R*-EXECUTOR.md）中所有“修正承诺”已逐条确认写入对应章节
✅ 无未关闭的 TODO/FIXME/待补充 标记
✅ state.json 中 phase=report_ready
```

**具体操作：**
1. 主 Agent 读取 `battle/BATTLE-R1-EXECUTOR.md`（如有 R2/R3 也读取）
2. 提取所有"接受"类回应中承诺的修正项
3. 逐条检查对应 GRV 章节文件，确认修正内容已写入
4. 如发现未修正项 → 发回 GRV 子Agent 执行修正，修正完成后重新检查
5. 全部确认 → 更新 state.json phase=report_ready → 启动合并

**禁止跳过此检查直接合并。违反此规则是 P0 级错误。**
