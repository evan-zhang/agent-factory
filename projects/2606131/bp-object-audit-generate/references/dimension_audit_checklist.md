## Part 2：7 维度操作化定义

### 维度 1：层级边界 (Level boundary)

**必须检查的字段**（按顺序）：
1. 目标编码前缀（G=集团 / A=中心 / D=部门 / 无前缀=关键个人）
2. 目标名称措辞（含"集团""中心""部门""个人"等层级关键词）

**检查动作**：
1. 取目标编码前缀 → 与当前审计层级比对
2. 扫名称关键词 → 标记跨层级措辞
3. **集团-中心边界上提/分流检查**（v1.5 §5.1.2/5.1.3 / R17/R18/R19/R42）：若审计集团层 BP 对象，逐条检查是否满足 5 个上提触发条件之一（见 `core_rules.md` § 3）；若属于 6 类分流条件之一（中心工作法/内部流程/部门分工/具体项目动作/工具建设细节/过程复盘动作）→ ❌（不应出现在集团层）

**通过条件（✅）**：编码前缀与审计层级一致，名称无跨层级措辞

**需确认条件（⚠️）**：名称含跨层级措辞但编码前缀正确

**不通过条件（❌）**：编码前缀与审计层级不一致

**与其他维度的边界**：
- 编码归属归维度1；语义表达问题归维度2
- "目标是否结果态"归维度2，不归维度1

---

### 维度 2：OKR 语义 (OKR semantics)

**必须检查的字段**（按顺序）：
1. 目标名称/成果名称（是否结果态描述）
2. KR 结构（是否有 KR 编号或 KR 子项）
3. 关键举措名称（是否路径描述，含实施动词）
4. 文本是否含 HTML 标签残留（`<h3>`, `<h4>`, `<br>`, `<p>`, `<div>` 等）

**检查动作**：
1. 目标：判断是否结果态（"实现""建立""跑通"是结果态；"推进""开展"是动作）
2. 成果：判断是否有 KR 子结构
3. 举措：判断是否为路径描述（动词开头 + 时间节点）
4. 扫全文 HTML 标签残留
5. **过程型表达扫描**（v1.5 §9 / R01）：成果层出现以下动词 → ⚠️（要求补充最终输出对象/验收条件）；目标层出现 → ❌；关键举措层可保留动作表达（不直接判错）：
   - 建立 / 启动 / 完善 / 构建 / 推动
   - 优化 / 跟踪 / 预警 / 复盘
6. **AI 数字化/外来概念扫描**（v1.5 §9.1/9.2 / R25/R26/R27）：检测未本地化的外部术语（如 FDE、AI Agent、Dashboard 直接作为 BP 标题）→ ⚠️，要求改写（详见 `core_rules.md` § 2 外来概念本地化）

**通过条件（✅）**：语义链完整 + 无 HTML 标签残留

**需确认条件（⚠️）**：语义链完整但有 HTML 标签残留

**不通过条件（❌）**：目标不是结果态 / 成果缺 KR / 举措不是路径描述

**与其他维度的边界**：
- HTML 标签残留归维度2（**不归**维度7）
- 语义深度不足归维度2（不归维度3）
- 目标编码前缀错归维度1（不归维度2）

---

### 维度 3：成果验收 (Outcome acceptance)

**必须检查的字段**（按顺序）：
1. 每个成果的 `measureStandard` 字段
2. 成果名称是否含可验证交付物

**检查动作**：
1. 逐个成果检查 measureStandard 是否非空
2. 验证 measureStandard 是否含量化要素（数字 + 单位 + 时间窗口）
3. 识别可验证交付物（制度文件、平台上线、报告、白皮书、SOP 等具名交付物）
4. **成果输出对象检查**（v1.5 §三 / R36/R37）：每个成果是否标注了主输出对象（8 类之一，见 `core_rules.md` § 2）；缺主输出对象 → ❌；过程证据误写为输出对象 → ❌

**通过条件（✅）**：全部成果有量化门槛或可验证交付物

**需确认条件（⚠️）**：部分成果 measureStandard 非空但量化要素不全

**不通过条件（❌）**：任一成果 measureStandard 为空且无可验证交付物

**与其他维度的边界**：
- 验收条件缺失归维度3；口径定义模糊归维度4
- `measureStandard=null` 主归维度7（不归维度3）
- 联动：若 measureStandard=null 导致 AI 判灯无依据，维度5 应同时记录 ⚠️

---

### 维度 4：口径对齐 (Definition alignment)

**必须检查的字段**（按顺序）：
1. 数值字段（百分比、分数、金额）的分子分母定义
2. 时间窗口定义（YTD、季度、月度）
3. 参照基准（"75 分位"、"行业 TOP3"、"YTD"）

**检查动作**：
1. 扫所有数值表述 → 检查分子分母是否定义
2. 检查 YTD/季度/月度等时间口径是否说明
3. 检查参照基准是否给出数据源
4. **审查主体与标准检查**（陈舒婷 Q5 补充）：若衡量标准涉及审查/评价，需明确审查主体（谁审查）和审查标准（怎么判）
5. **跨周期兑现检查**（陈舒婷 Q6 补充）：检查所有奖励、兑现、发放动作的时间节点是否在 BP 计划周期内；超出周期的需确认

**通过条件（✅）**：所有数值有明确口径 + 参照基准 + 审查主体明确（如涉及）+ 兑现时间在周期内

**需确认条件（⚠️）**：
- 存在"常态化运行""全覆盖"等无量化定义的表述
- 存在跨周期兑现动作（兑现时间超出 BP 计划周期）

**不通过条件（❌）**：
- 关键数值无分子分母且无参照基准
- 审查主体未定义（适用于涉及审查/评价的衡量标准）

**与其他维度的边界**：
- 数值口径模糊归维度4；验收条件缺失归维度3
- "待完善""待补充"等占位符归维度7（不归维度4）

---

### 维度 5：证据路径 (Evidence path)

**必须检查的字段**（按顺序）：
1. `cascade mode`（承接方式）
2. `downTaskList`（下游任务列表）
3. `taskUsers`（参与人列表）
4. 过程证据描述（文字）
5. 证据频率（月度/季度）

**检查动作**：
1. **先查承接方式**（这是判定前置条件，评审 Q4 修复点）：
   - 承接方式 = "不下拆" 或 "轻量跟踪"：`downTaskList=[]` → ✅ 合理，不下拆
   - 承接方式 = "完整 BP 承接" 或 "成果责任派发"：`downTaskList=[]` → ⚠️（单一路径，评审 Q2 修复）
   - 承接方式字段缺失：→ ❌（证据路径未知）
2. 检查 taskUsers 是否非空
3. 检查过程证据描述是否含"系统可查""月度报告"等可追溯描述
4. 检查证据频率是否明确
5. **字段名映射降级**（评审 Q3 修复）：若源文件字段名不是 `downTaskList`/`taskUsers`，记录"字段名 X 未找到，可能为 Y"；找不到的字段标 📊 进入问题队列
6. **承接方式唯一性检查**（v1.5 §7.3 / R44）：扫描承接方式字段，出现混写模式判 ❌：
   - "X + Y" 或 "X 与 Y" 并列（如 "成果责任派发 + 协同留痕"）
   - 字段出现"和"/"及"/"同时"/"并"等连接多个承接方式
   - 一条关键举措只能有一个主承接方式；其他参与/协同/输入关系写入"下级承接对象/分工"字段
   - **AI 不得直接冻结承接方式**：必须先输出判定卡（v0.2 cascade_decision_card.md），用户确认后才能进确认稿/正式 BP
7. **下级承接对象/分工必显检查**（v1.5 §7.3.1 / R29/R33/R34）：每条关键举措必须显式写出下级承接对象/分工（不得留空或写模糊词）。模糊词扫描（❌ 不得通过）：“各中心” / “相关部门” / “相关主体” / “O9相关主体”（O9 未生成时）/ “信息化相关团队” / “多部门协同” / “经营责任中心”（未反写时）/ 不填
8. **占位词反写检查**（v1.5 §7.4 / R39/R40/R41）：完整复盘版/关闭版/总复盘版必须反写占位词：
   - 各O主责中心 → 具体中心名
   - O9相关主体 → "待O9确认：数据安全、权限管理、隐私保护、AI治理和合规风控归口主体"
   - 信息化相关团队 → 信息技术部/经营管理中心数据管理部/玄关健康
   - 经营责任中心（未反写时）→ 深康/德镁/维盛/院外业务中心
   - 经营责任中心 vs 经营管理中心混淆 → 见 `core_rules.md` § 4 定义

**兼容性说明**：v0.1.9 新增字段 `下级承接对象/分工` 作为升级字段。源文件中旧字段名 `下级承接对象` 仍可读取，映射到同一语义层；旧字段存在时继续读取，但按 v0.1.9 要求检查"分工/不下拆理由"是否完整。

**通过条件（✅）**：承接方式 + 三个字段 + 频率全部齐备 + 承接方式唯一无混写 + 下级承接对象/分工已显式写出且无模糊词

**需确认条件（⚠️）**：
- 仅有文字描述"系统可查"但无字段支撑
- 承接方式 = 完整 BP 承接 但 downTaskList 为空数组（单一路径，评审 Q2 修复）

**不通过条件（❌）**：
- 承接方式字段缺失
- 承接方式 = 完整 BP 承接 + downTaskList=[] + 无文字证据描述
- 三个核心字段全部为空

**与其他维度的边界**：
- `downTaskList=[]` 归维度5（不归维度7）
- `measureStandard=null` 主归维度7；但若影响 AI 判灯，本维度应记录 ⚠️（联动规则，评审 Q3 修复）
- 责任人缺失归维度6（不归维度5）

### 维度 6：单主责 (Single owner)

**必须检查的字段**（按顺序）：
1. 每个举措的承接人/责任人字段
2. 每个举措的协办人字段
3. 多举措人员配置对比

**检查动作**：
1. 逐举措检查责任人是否非空 + 唯一
2. 扫所有举措 → 对比承接人配置
3. 检查主责与协办角色是否可清晰区分

**通过条件（✅）**：每条举措有唯一主责 + 协办角色清晰

**通过（边界情况，评审 Q5 修复）**：多举措承接人完全相同，但每条举措仍有**明确的主责和协同角色区分**（如主责顺序不同、协办人数不同、有辅助说明）→ 仍判 ✅

**需确认条件（⚠️）**：
- 多个举措承接人完全相同 + 主责/协办**无法明确区分**（无顺序、无辅助说明、协办人数完全相同）
- 责任人字段为空但有协办人

**不通过条件（❌）**：
- 责任人字段为空且无协办人
- 同一举措列出多个主责（不是协办）

**与其他维度的边界**：
- 责任人配置问题归维度6；证据链问题归维度5
- 主责与协办混用归维度6（不归维度2）

---

### 维度 7：冻结规则 (Freeze rules)

**必须检查的字段**（按顺序）：
1. `measureStandard` 字段值
2. 文本占位符（"____", "***", "TODO", "TBD"）
3. 待完善 / 待补充 / 待确认 等表述
4. 问号指标（"完成率 ≥ ?"）

**检查动作**：
1. 逐个成果检查 measureStandard 是否为 null
2. 全文扫描占位符
3. 扫待完善/待补充等表述（注意：BP-R<N> 已确认规则中的"待确认"标记不在此列）
4. 扫问号指标

**通过条件（✅）**：零违规

**不通过条件（❌）**：
- 任一 measureStandard = null
- 含占位符 / 待完善 / 待补充（除 BP-R 已确认规则外）
- 含问号指标
- HTML 标签残留**不**归此维度（归维度2）

**与其他维度的边界**：
- `measureStandard=null` 主归维度7（不归维度3）
- `downTaskList=[]` 归维度5（不归维度7）
- HTML 标签残留归维度2（**不**归维度7）

---

## Part 3：使用方式

1. 执行审计时按维度顺序（1→7）逐个调用本文件
2. 每个维度按"必须检查的字段"顺序读取数据
3. 按"检查动作"执行判定
4. 按"通过/需确认/不通过条件"给出结论
5. ⚠️ 和 ❌ 必须进入问题队列
6. 📊 数据缺失本身就是问题
7. 维度间边界冲突时，参考本文件"与其他维度的边界"判定归属
