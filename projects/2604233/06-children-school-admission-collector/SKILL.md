# 06 Children School Admission Collector

用于执行“子女上学”政策采集，不用于做最终裁决，也不替用户合并冲突口径。

## 能力边界

- 本 skill 只负责 `搜索检索`、`打开并采集页面内容`、`写入固定 Markdown 文件`
- 不负责政策裁决、资格判断、建议办理路径、电话咨询或替用户解释最终适用结论
- 不输出 3 个指定 Markdown 文件之外的独立文件
- 只写入本模块固定输出目录；不得读取、修改或合并其他 6 个 skill 的输出目录

## 何时使用

- 用户给出了 `目标城市` 和 `年份/时间范围`，或需要按城市和年份采集子女上学政策
- 用户希望采集 `子女入学` 模块，而不是笼统总结教育政策
- 用户要求逐渠道检索、逐指标记录、保留完整来源追溯，并固定输出 3 个 Markdown 文件

## 必要输入

- `目标城市`
- `年份/时间范围`

缺少任一必要输入时，先要求补齐；不要用 `2026年`、当前年份或记忆自动补齐。

执行参数：

- `模块`：`子女入学`
- `年份/时间范围`：使用用户输入值，必须写入 MiniMax 查询词、来源明细和缺口记录；不得擅自替换
- `人群`：`在职员工`
- `场景`：`非本地户籍或跨区域家庭子女在目标城市接受义务教育及参加中考/高考条件采集`
- `内置 reference`：[references/06-子女上学.md](references/06-子女上学.md)
- `固定输出目录`：`./七大政策/{目标城市}/06-子女上学/`

## 先读什么

1. 先打开本 skill 内置 [references/06-子女上学.md](references/06-子女上学.md)
2. 从内置 reference 提取：
   - 渠道顺序、渠道编号、渠道名称、渠道类型、主要用途
   - 标签库和关键词建议
   - 模块说明与输出约束
3. 根据 `目标城市` 和 `年份/时间范围` 将 `{城市}`、`{CITY}`、`{年份}`、渠道占位和检索词模板替换为目标城市与指定年份口径
4. 如用户额外提供城市材料，只能作为补充线索；不得替代本 skill 的固定指标、输出结构和内置 reference 渠道顺序

## SKILL.md 与 references 配合

- `SKILL.md` 是执行契约：决定输入、指标、输出文件、来源字段、追溯要求和禁止事项
- `references/06-子女上学.md` 是本模块内置模块文档：提供标签库、指标关键词映射、渠道模板和检索词扩展
- 执行顺序固定为：先读内置 reference，再根据目标城市和渠道模板定位真实页面，最后开始 MiniMax 检索与页面采集
- 不要求用户提供额外模块材料；如用户额外提供，只作为补充线索，不改变本 skill 的固定输出目录、指标和表头

## 检索工具要求

- 必须使用 `MiniMax web_search (via mcporter call minimax.web_search)`（MiniMax 搜索）进行充分检索
- 如果当前环境未安装或不可调用 `MiniMax web_search (via mcporter call minimax.web_search)`，停止执行并说明缺少检索技能；不要改用记忆、未打开页面或普通摘要替代
- 对内置 reference 中的每个渠道模板，按渠道顺序分别构造 MiniMax 查询，不得只做全网泛搜
- 每个渠道至少围绕该模块核心关键词和未覆盖指标补充查询；优先使用 `site:{渠道域名} + 指标关键词 + 年份/城市`
- 每个 MiniMax 查询都必须包含 `目标城市` 和 `年份/时间范围`；确需检索无年份政策原文时，也要补充一轮不含年份的查询并在备注说明
- MiniMax 用于发现和定位候选页面；采集内容必须来自已打开页面
- MiniMax 摘要、搜索结果片段或未打开页面不能当作官方原文
- 候选页面必须实际打开并验证可访问；`404`、打不开、空白页、错误页、登录拦截页、只有搜索结果而无正文的页面，不得作为采集来源
- `页面URL` 必须填写最终可打开的完整 URL；如 MiniMax URL 发生跳转，`页面URL` 写最终 URL，原始 MiniMax URL 写入 `参考URL` 或 `备注`
- 每条进入 `02-来源明细表.md` 的来源，都必须在 `备注` 写明 `URL验证：已打开/可读`，并记录 MiniMax 搜索结果的标题、排名或摘要片段
- 每条采用或未找到关键内容的记录，都要把实际 MiniMax 查询词写入 `检索关键词`

## 固定采集指标

1. 非本地户籍入学是否要社保（硬性门槛 vs 加分项）
2. 社保缴纳年限要求（多少年/个月）
3. 居住证是否必须（父母单方/双方）
4. 各区/街道执行标准差异
5. 房产 vs 租房在入学积分上的差异
6. 中考是否有社保要求及年限
7. 高考是否有社保要求及年限
8. 是否有上学时间节点要求（如：报名前须已在缴）
9. 优质学校对社保年限是否有梯度要求

## 专项采集约束

- 必须区分：
  - `义务教育入学`
  - `中考报名资格`
  - `高考报名资格`
  三者必须分别记录，不得混写
- `是否要社保` 必须区分：
  - `硬性门槛`
  - `积分项/加分项`
  - `推荐条件`
  如果只是积分项，不得写成硬性要求
- `社保缴纳年限要求` 必须明确：
  - `连续缴纳还是累计缴纳`
  - `父母一方还是双方`
  - `需缴满多少年/多少月`
  - `截止到哪个时间点`
- `居住证是否必须` 必须区分：
  - `父母单方即可`
  - `父母双方都需具备`
  - `儿童本人是否需要办理相关证件`
- `各区/街道执行标准差异` 必须优先保留 `区教育局`、`街道办`、`区招生简章` 等具体来源，不要只停留在市级原则
- `房产 vs 租房` 必须尽量区分：
  - `是否都可申请`
  - `积分差异`
  - `排位顺序差异`
  - `学位锁定、学区占用等附加规则（如找到）`
- `中考 / 高考是否有社保要求及年限` 必须明确：
  - `是学籍要求`
  - `还是监护人社保要求`
  - `还是户籍/居住证/学籍年限共同要求`
- `时间节点要求` 必须尽量明确到：
  - `报名前`
  - `信息采集前`
  - `当年某月某日之前`
  - `连续缴纳截至报名月`
- `优质学校梯度要求` 如果没有公开明确规则，不得推测；只能记录公开积分办法、排序办法、学位紧张提示等
- 如政策区分以下情形，必须拆开记录，不能混写：
  - `幼升小 / 小升初`
  - `公办 / 民办`
  - `户籍生 / 非户籍生`
  - `有房 / 租房`
  - `市级政策 / 区级实施细则`
- 没有明确政策原文时，不得推测，直接写 `未找到明确依据`

## 执行规则

- 以内置 reference 的渠道模板和顺序为准；不要自行重建无关渠道清单
- 必须按渠道顺序逐个检索；不要跳过
- 必须围绕 9 个指标逐项采集；不要只做摘要
- 同一指标允许保留多个来源，不强行合并成单一结论
- 每条采集结果都必须来自已打开页面；不要把未打开页面、搜索摘要或记忆当来源
- 官方优先；辅助来源可以保留，但必须标明来源级别
- 区级政策、区教育局招生方案、区积分入学办法、区报名系统公告优先级高于泛泛媒体总结
- 如果只拿到搜索引擎摘要但页面无法打开，不能作为来源写入 `01-指标汇总表.md` 或 `02-来源明细表.md`；只能在 `03-缺口与待补充.md` 记录该 URL、搜索摘要和失败原因
- `搜索快照` 只用于标注已验证可打开页面对应的搜索结果上下文，不能单独支撑指标数据；如果是二手转引，来源类型或备注写明 `二手转引`
- 本任务重点是“尽可能完整采集”，不是“综合判断谁最权威”
- 找不到明确内容时，写 `未找到明确依据`；如果缺的是官方依据，则写 `未找到明确官方依据`
- 不做最终政策裁决，不写“综合来看”“因此可判断”之类结论句

## 建议工作流

1. 建立来源台账
   - 建议来源编号格式：`{城市缩写或模块缩写}-{渠道编号}-{三位流水号}`
2. 逐渠道检索
   - 优先使用内置 reference 给出的检索词模板
   - 对每个命中页面，先打开正文，再记录完整元数据
   - 对存在区级差异的城市，优先补齐区教育局、区政府、街道或报名系统来源
3. 逐指标抽取
   - 抽取原文关键句、明确条件、适用对象、适用学段、适用区域、前置条件、例外情形
   - 明确该来源支撑的是义务教育、中考还是高考
   - 记录该来源支撑的具体指标编号
4. 生成交付文件
   - 只输出 3 个 Markdown 文件
   - 文件名、字段顺序、表头必须固定一致
   - 不额外输出检索日志、关键词矩阵、来源卡片等独立文件

## 输出文件

### 1. `01-指标汇总表.md`

- 固定按 9 个指标输出，一行对应一个指标
- 只呈现“当前采集到的信息”，不做最终裁决
- 每一行都要带来源编号、来源标题、来源 URL
- 同一个单元格内多个来源用 `<br>` 分隔
- 如果没有明确官方依据，明确写 `未找到明确官方依据`
- 若存在明显区级差异，必须在“当前采集到的信息”或“关键条件”中明确提示

使用以下固定表头：

```md
| 指标编号 | 采集指标 | 员工视角问题 | 当前采集到的信息（不做最终裁决） | 关键条件/年限/节点/积分差异 | 适用对象/适用学段/适用区域 | 官方来源编号 | 官方来源标题 | 官方来源URL | 补充来源编号 | 补充来源URL | 是否存在缺口 | 缺口说明 |
|---|---|---|---|---|---|---|---|---|---|---|---|---|
```

### 2. `02-来源明细表.md`

- 一行只对应“一个来源支撑一个指标”
- 同一个来源支撑多个指标时，必须拆成多行
- 必须完整保留来源信息，确保能回溯
- `页面URL` 必须是已打开验证可访问的最终 URL；404、打不开、跳转错误或不可读页面不得入表作为来源
- 原文摘录保留 1-3 句关键句
- “提取到的数据内容”尽量结构化、可复用

使用以下固定表头：

```md
| 指标编号 | 采集指标 | 来源编号 | 来源渠道名称 | 来源类型 | 来源级别 | 页面标题 | 页面URL | 参考URL | 发布日期/更新时间 | 抓取时间 | 检索关键词 | 适用对象/适用学段/适用区域 | 原文摘录 | 提取到的数据内容 | 备注 |
|---|---|---|---|---|---|---|---|---|---|---|---|---|---|---|---|
```

### 3. `03-缺口与待补充.md`

- 只记录存在关键缺口的指标
- 已完整找到的指标不要写进来
- 如果 9 项都没有缺口，仍然输出该文件，但只保留一行 `无缺口` 记录

使用以下固定表头：

```md
| 指标编号 | 采集指标 | 未找到的关键内容 | 已检索渠道/来源编号 | 已使用关键词 | 当前能确认到的边界 | 建议后续补查方向 | 备注 |
|---|---|---|---|---|---|---|---|
```

## 填写提醒

- `01-指标汇总表.md` 的“当前采集到的信息”只写事实，不写综合判断
- `01-指标汇总表.md` 的“关键条件/年限/节点/积分差异”只写页面中明确出现的条件；没有就写 `未找到明确数值`
- `01-指标汇总表.md` 的“适用对象/适用学段/适用区域”要明确写义务教育/中考/高考、本地或非本地户籍、有房或租房、具体区街道等，不要混写

- `02-来源明细表.md` 中同一来源支撑多个指标时要拆行
- `03-缺口与待补充.md` 只写真实缺口，不要把“还想补更多”也算进去
- 所有 URL 必须写完整
- 交付前检查 9 个指标是否都有一行
