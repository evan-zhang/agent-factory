## 一、收到链接后的决策树

```
收到链接
    │
    ├── 是否是 YouTube？
    │   └── YES → 进入 YouTube 处理流程

    └── 其他链接
        └── 进入通用网页处理流程
```

## 二、第一步：抓取内容

### 2.1 今日头条文章

**首选方案：curl + jina.ai 代理**

```bash
curl -sL --max-time 60 "https://r.jina.ai/https://m.toutiao.com/is/xxxxx/" 2>&1
```

**思考过程：**
1. 今日头条有反爬机制，直接 curl 可能超时或被拦截
2. jina.ai 是内容代理服务，可以绕过反爬，返回干净的 Markdown
3. `--max-time 60` 防止无限等待

**常见问题：**
| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 超时（exit code 28） | 今日头条限制或网络问题 | 换 Web Search |
| 返回乱码 | 编码问题 | 忽略，依赖 Web Search 补充 |

**备用方案：Web Search**
```bash
web_search(query="<关键词>")
```

**思考过程：**
- Web Search 用 Perplexity，可以理解上下文
- 即使抓不到完整内容，也能通过搜索获取关键信息
- 搜索时用项目名 + 功能关键词

### 2.2 YouTube 视频

**第一步：获取基本信息**

```bash
yt-dlp --print title --print duration --print uploader "<YouTube链接>" 2>&1
```

**思考过程：**
- `--print title` 获取视频标题（用于判断内容）
- `--print duration` 获取时长（用于判断是否值得转录）
- `--print uploader` 获取频道名（用于判断来源可靠性）
- 只需要基本信息，不需要下载，用 `--print` 而不是完整下载，节省时间

**第二步：判断是否需要转录**

| 时长 | 决策 | 理由 |
|------|------|------|
| < 5 分钟 | 通常不需要 | 信息密度低，可以直接总结 |
| 5-15 分钟 | 询问用户 | 标准长度，视内容价值决定 |
| > 15 分钟 | 通常需要 | 信息量大，值得完整转录 |

**第三步：下载音频 + 转录**

```bash
# 下载音频
yt-dlp -x --audio-format mp3 -o "temp/video-title.%(ext)s" "<YouTube链接>"

# Whisper 转录（中文）
whisper temp/video-title.mp3 --language Chinese --output_format txt --output_dir temp/
```

**思考过程：**
1. `yt-dlp -x` 提取音频而非视频，文件更小
2. `--audio-format mp3` 转换为 MP3 格式
3. Whisper 需要音频文件，中文用 `--language Chinese`
4. 输出到 `temp/` 目录，方便清理
5. 转录完成后删除临时文件 `rm temp/*.mp3 temp/*.txt`

### 2.3 其他网页

**首选方案：curl + jina.ai**

```bash
curl -sL --max-time 30 "<URL>" 2>&1
```

**备用方案：web_fetch**

```bash
web_fetch(url="<URL>", extractMode="markdown", maxChars=50000)
```

**思考过程：**
- `web_fetch` 是 OpenClaw 内置工具，直接返回 Markdown
- 适合结构化的网页（文档、博客等）

## 三、第二步：判断内容类型和质量

### 3.1 内容分类

| 类型 | 特征 | 处理方式 |
|------|------|---------|
| **开源项目介绍** | GitHub、Star、功能描述 | 重点分析架构、技术栈、团队背景 |
| **新闻/资讯** | 事件描述、快速传播 | 提取关键信息，不写长报告 |
| **教程/讲解** | 步骤说明、如何使用 | 提取步骤，可操作执行 |
| **对比分析** | A vs B、优劣势 | 重点做对比表格 |
| **观点/评论** | 个人看法、趋势判断 | 记录观点，不做深度分析 |

### 3.2 质量判断

**高质量内容的特征：**
- 有具体数据（Star 数、融资额、用户数）
- 有技术细节（架构、代码、算法）
- 有对比分析（vs 其他项目）
- 有实际案例（落地场景）

**低质量内容的特征：**
- 只有标题党
- 没有实质内容
- 信息过时（1 年前）

### 3.3 是否值得写报告的判断

| 标准 | 值得写报告 | 不值得写报告 |
|------|-----------|--------------|
| **新颖性** | 新项目/新功能 | 已知项目的一般更新 |
| **信息量** | 有技术细节、数据 | 只有概述性描述 |
| **与我们相关** | 和已有调研有联系 | 完全独立无关 |
| **可操作性** | 有实际价值 | 纯概念介绍 |

## 六、第五步：存档 + 发送

### 6.1 存档到 Obsidian

```bash
OBSIDIAN="/Users/evan/Library/Mobile Documents/iCloud~md~obsidian/Documents/github分享"
mkdir -p "$OBSIDIAN/AI调研/YYYY-MM-DD"
cp <报告.md> "$OBSIDIAN/AI调研/YYYY-MM-DD/"
```

**思考过程：**
- Obsidian 有 iCloud 同步，Evan 可以随时访问
- 按日期分目录，方便查找
- 保留原始文件名

### 6.2 Git Commit

```bash
cd /Users/evan/.openclaw/gateways/life/state/workspace-life
git add <报告.md>
git commit -m "docs: <简短描述>"
```

**Commit message 格式：**
```
docs: <项目名> <一句话描述>
```

**例子：**
```
docs: MiroFish 群体智能预测引擎调研（45.9K Stars）
```

### 6.3 发送到 Discord

```python
message(
  action: "send",
  channel: "discord",
  target: "channel:1488700354718859324",
  message: "📊 <报告标题>\n\n<简短摘要>",
)
```

**思考过程：**
- 先发摘要，让用户快速了解内容
- 如果用户需要详细信息，再发完整报告附件
- 但 Discord 附件功能有 bug，无法正常发送

**Discord 发送的已知问题：**
- `media` 参数发送文件附件时报 Content-Type 错误
- 解决方案：把文件内容作为 Markdown 消息发送

## 七、特殊情况处理

### 7.1 链接打不开

**今日头条链接打不开：**
1. 换 `https://r.jina.ai/` 代理（timeout=60s）
2. 仍然失败 → Playwright + wait_for_timeout(5000) 等待动态内容
3. 仍然失败 → 用 Web Search 搜索内容

**YouTube 链接无法下载：**
1. 检查是否是短链接格式（需要展开）
2. 尝试直接访问看是否需要登录
3. 仍然失败 → 跳过或告知用户

### 7.2 内容质量差

**文章只有标题党，没有实质内容：**
- 不写报告
- 简短告知用户内容价值低
- 记录"无效链接"避免重复处理

**信息过时（1 年以上）：**
- 不写报告
- 告知用户内容可能已过时

### 7.3 重复内容

**之前已调研过的项目：**
- 简要补充新信息
- 指出"与之前调研的区别"
- 不重复写完整报告

### 7.4 沙盒环境限制

**在沙盒中无法抓取内容：**
- 检查网络出站规则
- 检查 DNS 配置
- 检查 SSL 证书

## 十、常见错误与避免

| 错误 | 错误原因 | 避免方法 |
|------|---------|---------|
| Star 数不准确 | 引用过时数据 | 每次都搜索最新数据 |
| 重复发送报告 | 发送两次 message | 发送前检查 |
| 文件路径错误 | 复制粘贴路径 | 用 tab 补全 |
| commit message 模糊 | 没写清楚 | 用"docs: 项目名 一句话描述"格式 |
| 报告结构混乱 | 没有模板 | 按照 SOP 模板写 |
