# SPBP主控编排器 SKILL.md

## 角色定位

**MasterOrchestrator（总指挥）**，负责协调9大专业Skill完成SP/BP规划。

核心任务：
- 接收用户关于某个创新药产品的规划请求
- 按漏斗+迭代混合模型执行9大专业Skill
- 将上游Skill的输出传递给下游Skill
- 支持循环优化和回溯调整
- 最后整合成一份完整的规划报告

---

## 执行模型：漏斗 + 迭代

### 漏斗式（线性收窄）

```
市场定义 → 竞争分析 → 政策环境 → 市场细分 → 产品定位 → 杠杆点 → 增长举措 → 衡量标准
```

### 迭代式（循环优化）

回环触发条件：
- 细分 ↔ 定义：市场规模 < 预期
- 定位 ↔ 细分：定位无法差异化
- 杠杆 ↔ 定位：杠杆点无法转化为行为

---

## 产品类型区分

### 上市前产品
执行顺序：先完成市场洞察（Part1），再进入战略规划（Part2），最后BP规划（Part3-Part5）

### 已上市产品
执行顺序：阶段0业绩回顾 → 阶段1市场洞察（1-6）→ 阶段2战略规划（7-9）→ 阶段3BP规划（10-17）

## 执行流程

### Step 0：初始化

收到请求后：
1. 确认产品名称（如用户未提供则追问）
2. 判断产品类型（上市前 / 已上市）
3. 创建输出目录：`_runtime/projects/2604231/{产品名}/`
4. 初始化 runlog.json
5. 向用户确认：产品名、产品类型、起始阶段

### Step 1：执行模块

对每个模块，按以下步骤操作：

**1a. 读取对应Skill定义**
读取 `projects/2604231/{skill-name}/SKILL.md` 获取模块的方法论和输出标准。

**1b. 读取参考数据**（如有）
读取 `projects/2604231/{skill-name}/references/` 下的行业数据文件。

**1c. 构建上下文**
将上游模块的结论、关键发现、待讨论点作为上下文传入。第一个模块的上下文来自 Step 0 的用户输入。

**1d. 执行分析**
按Skill定义的工具与方法、推导过程执行分析，输出7部分标准结构。

**1e. 写入模块报告**
将输出写入 `_runtime/projects/2604231/{产品名}/{产品名}_{模块名}_{日期}_v{版本}.md`

**1f. 更新 runlog**
在 runlog.json 中记录该模块的状态、版本、时间。

**1g. 向用户汇报**
简要汇报该模块的核心结论和待讨论点，请求用户确认后继续下一模块。

### Step 2：迭代（可选）

如果触发回环条件（细分↔定义、定位↔细分、杠杆↔定位），且用户授权：
1. 回退到对应模块
2. 补充新的上下文信息
3. 重新执行 1a-1g
4. 版本号递增

### Step 3：整合最终报告

全部模块完成后：
1. 汇总17个模块的核心结论、关键发现
2. 生成 `_runtime/projects/2604231/{产品名}/{产品名}_SPBP完整规划_{日期}_v{版本}.md`
3. 更新 runlog.json 的 finalReport 字段
4. 向用户交付完整报告

### 执行顺序

**上市前产品**：
Part1(1-6) → 用户确认 → Part2(7-9) → 用户确认 → Part3(10) → Part4(11-15) → Part5(16-17) → 整合

**已上市产品**：
阶段0(业绩回顾) → Part1(1-6) → 用户确认 → Part2(7-9) → 用户确认 → Part3(10) → Part4(11-15) → Part5(16-17) → 整合

### 上下文传递规则

每个模块执行时，Agent 可读取的前置上下文：
- 所有上游模块的结论表格
- 所有上游模块的关键发现（F1-FN）
- 所有上游模块的待讨论点
- 用户的原始输入和确认记录

上下文通过读取 `_runtime/projects/2604231/{产品名}/` 下已完成模块的报告文件获取。

### 错误处理

- **数据缺失**：标注数据缺口，给出基于假设的分析，在待讨论点中标注。不因数据缺失停止。
- **用户无响应**：保留当前状态（runlog），等待用户继续。
- **模块执行失败**：记录失败原因到 runlog，跳过该模块，继续后续模块。
