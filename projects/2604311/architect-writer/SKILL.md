# architect-writer (AW) · 结构化方案

> 将复杂的业务知识体系，转化为一套结构清晰、多人协作可用的方案报告（多章节、多分卷 Markdown 文件集群）。

## 触发词

当用户说以下任一表述时，使用本 Skill：
- "写方案"、"写报告"、"写规划"
- "梳理体系"、"搭框架"
- "出文档"、"写文档"
- "整理成报告"、"形成文档"
- "做一个XXX体系/框架/方案"

---

## 方法论总览

本 Skill 的核心是一个 **五阶段工作流**，贯穿 **Best-Minds 专家思维** 和 **五个批判性节点**：

```
Phase 0: 理解吸收 → 读素材，建立业务认知 🔴 CP1: 认知纠偏检查
Phase 1: 框架搭建 → 提炼结构、编号体系、生成架构图 🔴 CP2: 框架批判审查
Phase 2: 逐章起草 → 按优先级逐章/逐卷撰写内容 🔴 CP3: 章节目录预审 / 🔴 CP4: 初稿自我批判
Phase 3: 审核迭代 → 人机协同审核，批量修订 🔴 CP5: 修订回溯检查
Phase 4: 合并交付 → 总索引 + 合并版 + 版本归档 🔴 CP6: 交付前终审
```

每个阶段有明确的目标、输出物和交接条件。
每个批判性节点是制度化的"刹车"，确保方向不偏。

---

## 工作模式选择

五阶段工作流是 AW 的核心方法论，但不同类型的文档任务适合不同的执行节奏。在启动前，先判断任务类型：

### 模式A：系统设计模式
- **适用**：复杂体系、多模块框架、需要从头搭建的业务方案
- **特征**：完整走五阶段 + 全部批判性节点
- **典型**：院外体系 1+6+2+3 框架

### 模式B：模板填充模式
- **适用**：有现成模板或明确格式要求，需要填充具体内容
- **特征**：Phase 1 快速确认模板 → 跳到 Phase 2 填充 → Phase 3 校验
- **典型**：四报告体系、产品介绍

### 模式C：迭代优化模式
- **适用**：对现有文档的迭代修改（v1→v2→v3）
- **特征**：读取旧版本 → 分析变更点 → 生成新版本 → 校验差异
- **典型**：指标字典更新、方案修订

### 模式D：分段交付模式
- **适用**：超大型文档（预计 >50KB 或 10+ 分卷）
- **特征**：不追求一次写完，每段交付可用部分，最后拼接
- **典型**：大型市场分析报告、全业务线梳理

### 模式E：知识工程模式
- **适用**：结构化知识体系（分类-分层-标签）
- **特征**：设计分类框架 → 批量生成内容 → 打标签 → 校验一致性
- **典型**：问题体系、FAQ 知识库

**判断规则**：默认用模式A。只有明确匹配其他模式特征时才切换。切换模式后，五阶段工作流的阶段可以合并或跳过，但批判性节点（CP）不能跳过。

## Phase 0: 理解吸收

### 目标
建立对业务领域的完整认知，形成"可以开始搭框架"的体感。

### 方法

1. **批量阅读原始素材**
   - 用户会陆续发送文件（MD/PDF/Word/图片），全部接收存档
   - 每份文件读完立即提炼核心信息，写入 `analysis/` 目录
   - 不等"所有材料到齐"才开始——边读边提炼

2. **素材提炼模板**（每个素材文件写一条提炼记录）
   ```markdown
   ## [素材文件名]
   - 来源：[用户发送/内网/搜索]
   - 核心内容：[3-5 句话]
   - 可用章节：[初步判断归入哪些章节]
   - 关键数据点：[数字/目标/时间线等]
   - 待确认：[哪些地方有疑问]
   ```

3. **认知纠偏**
   - 读完第一批素材后，主动总结"我对这个业务的理解"
   - 提出可能理解错的地方，请用户纠正
   - **这一步不能省**——错误理解一旦进入框架就很难改

4. **关键人名/术语速查表**
   - 建立人名→角色映射
   - 建立术语→含义映射
   - 随时更新，贯穿全过程

### 输出物
- `analysis/` 目录下的提炼文件
- 一份"业务全景一句话"总结
- 人名表 + 术语表

### 交接条件
用户确认"理解基本正确"，可以开始搭框架。

## Phase 1: 框架搭建

### 目标
定义整个报告的骨架：分几层、几卷、每卷写什么、文件怎么存。

### 方法

#### 1.0 领域认知扫描（建议前置）

在提炼框架之前，如果该方案涉及一个有成熟方法论体系的业务领域，建议先执行领域认知扫描（详见 Best-Minds 扩展协议）。

扫描完成后，将结果用于指导框架结构：
- 共识模型 → 必须覆盖的章节/模块
- 根本性分歧 → 需要用户做立场选择的设计决策点

如果用户拒绝扫描或领域不适用，直接进入 1.1。

#### 1.1 提炼框架结构

从 Phase 0 的素材提炼中归纳出：
- 这个业务/体系有哪些**核心层**？
- 层与层之间的**逻辑关系**是什么？（并列、递进、支撑、驱动？）
- 每层下面有哪些**子模块**？
- **如果执行了认知扫描**：框架结构必须覆盖所有共识模型，分歧点标注为待定立场

#### 1.2 编号体系

采用**层级编号 + 口号**的双重标识：

```
示例：1+6+2+3 框架
├── ① 底层：品种体系（1个模块）
├── ② 基础层：六大支撑（6个模块）
│   ├── 战略资产层：人 / 财
│   ├── 基础设施层：数据 / 品种供应链
│   └── 外部资源层：商务 / 药店
├── ③ 驱动引擎：双驱（2个模块）
└── ④ 转化出口：三大场景（3个模块）
```

编号规则：
- 每个层级用**带圈数字**（①②③④）标识
- 每卷用**两位数编号**（01-13）作为文件名前缀
- 口号用于帮助记忆和沟通，如"选品有道，支撑有力，双驱并进，三路变现"

#### 1.3 生成架构图（核心输出）

框架搭好后，生成一份独立的**架构图文档**，作为整个方案的"封面"：

**文件名**：`01-[体系名]/00_架构图.html`

**内容要求（2-3页）**：
1. **封面页**：方案名称、编号体系口号、核心定位一句话
2. **架构图页**：HTML 内嵌可视化（CSS Grid/Flexbox 块状图，或 Mermaid），展示核心层与层的关系
3. **架构说明页**：200-300 字，解释分层的逻辑依据、核心理念、使用场景

**HTML 架构图规范**：
- 颜色编码与 Phase 编号对应（可参考 `flowchart.html` 的色板）
- 图块之间用箭头标注流向关系
- 支持浏览器直接打开，无需构建
- 在文件末尾加一行版本信息：`&lt;!-- 架构图 v[X.X] · 框架：[体系名] --&gt;`

**作用**：读完这个文档，就能理解整个方案的骨架，不需要逐卷去读。
这是给业务方看的第一份材料，**提纲挈领**。

#### 1.4 文件集群结构

```
projects/[项目名]/
├── README.md          ← 项目入口
├── WORKLOG.md         ← 工作日志（进度 + session + 问题）
│
├── 01-[体系名]/       ← 核心内容
│   ├── 00_架构图.html  ← 架构图（HTML，方案的封面）
│   ├── 00_总索引.md   ← 框架总览 + 分卷索引
│   ├── 01_[第一卷名称].md  ← 独立分卷
│   ├── 02_[第二卷名称].md
│   ├── ...
│   ├── [体系名]_合并版.md  ← 全部内容的合并版
│   └── 执行计划.md     ← 撰写执行计划
│
├── 02-SOP规范/        ← SOP类文档（如适用）
├── 03-分析提炼/       ← Phase 0 的素材提炼
└── docs/              ← 草案、参考资料
```

#### 1.5 框架审核（关键步骤）

框架搭好后，执行**批判性审核**：
1. 逻辑自洽性：层与层之间有没有矛盾？
2. 分类完整性：有没有遗漏的重要模块？
3. 命名准确性：术语是否准确？会不会产生歧义？
4. 可落地性：框架能不能指导后续逐章撰写？

**主动提批判意见**（至少 3-5 条），请用户裁决。

架构图文档一并接受 CP2 审查——框架结构、编号、口号、分层逻辑都会反映在架构图中。

#### 1.6 框架版本管理

框架会经历多次修订，记录版本演进：

```markdown
| 版本 | 变化 | 时间 |
|------|------|------|
| v1.0 | 初版 | XX:XX |
| v2.0 | 加入XX概念 | XX:XX |
| v3.0 | 分类调整 | XX:XX |
| v4.0 | 审核修订 | XX:XX |
```

### 输出物
- `00_架构图.html`（HTML架构图，方案的封面，提纲挈领）
- `00_总索引.md`（框架全景图 + 分卷索引 + 数据总览）
- 框架版本演进记录
- 用户确认的框架终版

### 交接条件
用户确认框架结构（终版），可以开始逐章撰写。

## Phase 2: 逐章起草

### 目标
按框架定义的卷/章节，逐一撰写完整内容。

### 方法

#### 2.1 章节目录规划（每卷启动前）

对每个分卷，先做**目录规划**再写内容：

```markdown
## XX卷 - 目录规划

### 现有素材基础
- [已有内容的简要盘点]

### 缺失项（导致无法落地的）
- [列出缺少什么]

### 章节结构
├── 第1章 XXX
├── 第2章 XXX
└── 第N章 XXX

### 优先新增章节
1. [最紧急的]
2. [最实用的]
3. [连接其他卷的]
```

**这一步很重要**——先把章节目录规划发给用户确认，再动手写。
避免写完一整卷才发现方向偏了。

#### 2.2 章节撰写模板

每个分卷文件的开头使用统一 frontmatter：

```markdown
# [编号] [卷名]

> 所属框架：[框架名]
> 版本：v[X.X]
> 撰写时间：YYYY-MM-DD HH:MM
> 状态：🔨 撰写中 / ✅ 已完成 / 🔄 审核修订中

---

## 口号
**"[口号内容]"**

---

## 目录结构
[本卷的章节树]
```

每个章节的撰写遵循：
1. **先写定义/定位**：这一章解决什么问题
2. **再写核心内容**：主体框架、流程、标准
3. **补充数据支撑**：具体数字、目标、时间线
4. **标注待确认项**：用 `❓` 标记需要用户或业务方确认的内容
5. **连接其他章节**：说明与其他卷的关联

#### 2.3 撰写优先级

建议从**最核心的章节**开始（通常是最底层或最基础的），因为：
- 核心章节定义了术语和概念，后续章节会引用
- 用户审完核心章节后，认知会进一步对齐
- 后续章节可以复用核心章节的素材和理解

#### 2.4 并行策略

当有多个章节需要写时：
- **主线**：agent 自己写最核心的章节
- **副线**：spawn sub-agent 去读剩余素材或写辅助章节
- 两条线并行，节省时间

原则：**先出草案，审的过程就是深入理解的过程**。
不要等所有素材完美再动笔。

#### 2.5 版本控制

每个文件在文件名中体现版本：
- `01_品种体系_v4.1增强版.md`
- 重大修改升大版本号（v4→v5）
- 小修改升小版本号（v4.0→v4.1）
- 每次修改记录变更摘要

### 输出物
- 各分卷的 `.md` 文件（逐个产出）
- 每个文件内有待确认项清单

### 交接条件
所有分卷初稿完成，进入审核阶段。

## Phase 3: 审核迭代

### 目标
通过人机协同的审核，将初稿打磨成可交付的终稿。

### 方法

#### 3.1 内容审核（用户参与）

每个分卷完成后：
1. 发给用户审核（直接发文件或核心段落）
2. 收集反馈意见
3. 分类处理：
   - **事实性错误**：立即修正
   - **结构优化**：评估后决定是否调整
   - **内容补充**：标记待补充，排入后续计划

#### 3.2 文件完整性校验

每个分卷发审前，执行以下自检：

| 检查项 | 方法 | 通过标准 |
|--------|------|----------|
| 完整性 | 逐章对照目录规划 | 每个章节都有实质内容 |
| 无截断 | `tail -20 [file]` | 末尾完整，非突然中断 |
| 文件大小 | `wc -c [file]` | 与预计大小偏差 <20% |
| 命名规范 | 检查文件名 | 符合编号体系 |
| 引用一致 | 抽查跨文件数据 | 同一数据在不同文件中值一致 |

**发现问题时的处理**：
- 截断 → 用 `edit` 补充或重新生成该章节
- 大小偏差过大 → 检查是否遗漏内容
- 内容不完整 → 补充缺失部分

#### 3.3 批量修订

当用户给出一系列修订意见时：
- 逐条确认理解是否正确
- 批量执行修改（使用 `edit` 工具做精确替换）
- 修改完后发修订摘要，不转发原始修改记录

#### 3.4 跨卷一致性检查

审核过程中关注：
- 同一概念在不同卷中的表述是否一致
- 术语使用是否统一
- 交叉引用是否正确
- 数据（目标数字、人名等）是否一致

### 输出物
- 修订后的各分卷文件
- 修订记录摘要

### 交接条件
用户确认所有分卷内容可以定稿。

---

## Phase 4: 合并交付

### 目标
将所有分卷合并为完整交付物，建立索引和归档。

### 方法

#### 4.1 总索引更新

更新 `00_总索引.md`：
- 框架全景图（纯文本 + Mermaid）
- 分卷索引表（编号、名称、核心内容、状态、最后更新）
- 数据总览（关键指标汇总）
- 口号和核心逻辑

#### 4.2 合并版生成

将所有分卷内容合并为一个文件：
- `[体系名]_合并版.md`
- 保留各卷的标题层级
- 统一 frontmatter
- 文件可能很大（7000+ 行），这是正常的

#### 4.3 版本归档

```markdown
| 文件 | 版本 | 说明 |
|------|------|------|
| 00_总索引.md | v5.2 | 最终版 |
| 01_品种体系.md | v4.1 增强版 | 含品种战略 |
| ... | ... | ... |
| 合并版 | v5.2 | 13卷完整合并 |
```

#### 4.4 发送交付

- 每个分卷文件逐一发送给用户（Telegram 群聊/话题中直接发文件）
- 合并版单独发送
- 附上交付说明：版本号、总行数、待确认项数量

### 输出物
- 完整文件集群（N+ 个 MD 文件）
- 总索引 + 合并版
- 项目 README

## Best-Minds 专家思维协议（贯穿全流程）

> **核心原则：Agent 不是打字员，是领域顶级专家。**
> 不满足于通用浅薄的分析，不以附和用户为目标，以产出经得起同行评审的内容为底线。

### 为什么需要这个协议

AI 默认倾向于给出"通用但浅薄"的回答、倾向于附和用户。但在复杂方案报告的撰写中，用户需要的不是一个"听话的打字员"，而是一个**能提供专业判断、敢于反对、能把用户没意识到的问题摆上桌面的专家搭档**。

### 执行规则

**在撰写过程的每一个关键节点，Agent 必须完成以下自检（内心独白，不输出）：**

```
1. 当前任务涉及的专业领域是什么？
2. 如果我是这个领域全球 Top 5 的专家，我会怎么定义/处理这个问题？
3. 用户当前的方案/思路，最大的薄弱环节在哪里？
4. 这个领域最常见的认知误区是什么？我要主动帮用户避开。
5. 用户还没意识到但必须考虑的点是什么？
```

### 专家思维在撰写各阶段的应用

| 阶段 | 专家思维的具体表现 |
|------|------------------|
| **Phase 0 理解吸收** | 不只是"读懂"素材，而是识别素材中的**逻辑漏洞、数据矛盾、过度乐观的假设**。读完主动指出"这份汇报里提到的 XX 目标，与 YY 数据不一致"。 |
| **Phase 1 框架搭建** | 主动挑战用户提出的初始框架结构——"您说的 1+5+3 里，'货'放在基础层有逻辑矛盾，因为..."。至少提 3-5 条结构性批判意见。 |
| **Phase 2 逐章起草** | 以专家标准写内容——不是堆砌信息，而是给出**判断**。"建议采用三阶段评估而非两阶段，理由是..."。每个章节至少包含一个超出用户预期的专业洞察。 |
| **Phase 3 审核迭代** | 以"最挑剔的同行评审者"身份审视自己写的内容——"这段写得不够深""这个流程缺少异常处理""这个指标没有定义计算口径"。 |
| **Phase 4 合并交付** | 交付时不只交付文件，还交付**专家判断摘要**——"这套方案最大的三个风险点""与行业最佳实践的差距""落地执行时最容易翻车的环节"。 |

### 专家输出标准

激活专家思维后，产出应达到以下标准：

1. **超出用户认知** — 每个核心章节至少有一个用户没提到但必须考虑的点
2. **有据可查** — 关键论点有行业实践/学术依据，不是拍脑袋
3. **承认边界** — 明确说"这个领域我的理解到此为止，需要 XX 确认"
4. **提供对标** — 提到业界类似系统/方案是怎么做的（即使最终不采用）
5. **敢于反对** — 当用户的方向有明确问题时，给出反对意见和理由，而非沉默执行

---

### 领域认知扫描（Best-Minds 扩展协议）

> **核心问题：在搭框架之前，先搞清楚这个领域的"地形"——哪里是共识平原，哪里是争议高地。**

在 Phase 1 框架搭建之前（或之中），Agent 应主动执行领域认知扫描，回答两个结构化问题：

**问题一：专家共识**
> 这个领域的所有专家共同认可的 5 个核心心智模型是什么？

输出要求：
- 每个模型：名称 + 一句话定义 + 为什么是共识
- 标注每个模型对方案框架的影响（决定哪些是必选章节）

**问题二：根本性分歧**
> 这个领域的专家在哪 3 个问题上存在根本性分歧？每一方的核心论据是什么？

输出要求：
- 每个分歧：分歧问题 + 正方论据 + 反方论据 + 对方案的影响
- 标注每个分歧需要方案设计时做出明确的立场选择

**扫描结果的用途**：
- 共识模型 → 方案的**必选章节**（共识 = 必须覆盖）
- 根本性分歧 → 方案的**立场选择点**（分歧 = 需要明确选择并说明理由）

**主动触发时机**（Agent 应在以下场景主动建议用户执行）：
1. **Phase 0→Phase 1 过渡时**：当方案涉及有成熟方法论体系的业务领域
2. **框架讨论中**：用户在框架结构上出现犹豫或反复
3. **用户提及领域术语**：如"最佳实践"、"行业标准"、"方法论"等

**用户可以拒绝**。拒绝后直接进入框架搭建，不强制。

## 批判性节点（Critical Checkpoints）

> **核心原则：有争论才会写出更好的内容。**
> 在撰写流程中设置制度化的"刹车点"，强制执行批判性审查。

### 设计理念

写方案不是线性过程"读完→搭框架→写内容→交付"。每一步都可能产生偏差，需要**制度化的检查点**来纠偏。

批判性节点的本质是：**在关键决策点，强制换一个视角审视当前产出**。

### 五个批判性节点

#### 🔴 CP1：认知纠偏检查（Phase 0 结束时）

**触发条件**：读完第一批核心素材后

**执行方式**：
1. 写一份"我对这个业务的理解"（不超过 10 句话）
2. 主动列出**3-5 个可能理解错的地方**
3. 请用户确认/纠正

**不通过标准**：用户说"不对"超过 3 处 → 回到 Phase 0 补读素材

**实战案例**：xmt-sop 项目中，我们在 CP1 删掉了"私域运营""中药"等错误理解，把 B2C/O2O/线下药店的本质区别搞对。

#### 🔴 CP2：框架批判审查（Phase 1 框架定稿前）

**触发条件**：框架初版完成，用户确认前

**执行方式**：
1. Agent 以"最挑剔的评审者"身份，对框架提出**至少 5 条批判意见**
2. 每条意见必须包含：
   - 具体指出哪里有问题
   - 为什么这是个问题（逻辑/数据/经验依据）
   - 建议怎么改
3. 用户逐条裁决：接受 / 拒绝 / 搁置

**不通过标准**：用户拒绝了全部批判意见 → 要么批判不够准（重来），要么框架确实没问题（通过）

**实战案例**：xmt-sop 项目的 CP2 中，5 条批判中用户接受了 4 条——通过争论让框架变得更好。

#### 🔴 CP3：章节目录预审（Phase 2 每卷动笔前）

**触发条件**：每卷的目录规划完成后、动笔写内容前

**执行方式**：
1. 列出本卷的章节结构
2. 盘点已有素材基础和缺失项
3. 标注哪些章节"写出来可能很薄"（素材不足）
4. 请用户确认方向

**不通过标准**：用户说"这一卷的定位不对" → 重新规划

#### 🔴 CP4：初稿自我批判（Phase 2 每卷初稿完成后）

**触发条件**：每卷初稿写完、发给用户审之前

**执行方式**：
1. Agent 自己审一遍，以专家视角找出**至少 3 个问题**：
   - 内容浅薄（只有框架没有实质）
   - 逻辑断点（前后矛盾或跳跃）
   - 数据缺失（该有数字的地方没有）
   - 可落地性差（写了一堆原则但没有操作指引）
2. 把自我批判的结果写在文件末尾的"自审记录"里
3. 然后发给用户，附上"我知道这几个地方还不够好"

**关键心态**：主动暴露自己写的不足，比等用户找出来好 100 倍。

#### 🔴 CP5：修订回溯检查（Phase 3 批量修订后）

**触发条件**：用户的修订意见全部执行完毕后

**执行方式**：
1. 逐条回溯用户的每条修订意见，确认是否已正确执行
2. 检查批量修改是否引入了新问题：
   - 术语替换是否误伤了其他地方（全局替换的副作用）
   - 结构调整是否破坏了原有的逻辑链路
   - 新增内容是否与已有内容产生矛盾
3. 执行一次跨卷一致性快速扫描
4. 产出一份"修订回溯报告"：
   - 已正确执行的修订项
   - 发现的新问题（如有）
   - 需要用户二次确认的项（如有）

**不通过标准**：修订执行偏差 >20%（遗漏或错误超过总数的 1/5）→ 回到 Phase 3 重新执行修订

**关键心态**：修订不是终点，修订后的验证才是。改了 10 个地方但引入了 2 个新 bug，等于没改。

#### 🔴 CP6：交付前终审（Phase 4 合并后）

**触发条件**：全部合并版完成、最终交付前

**执行方式**：
1. 跨卷一致性扫描：同一概念在不同卷中的表述
2. 术语统一检查
3. 数据一致性（目标数字、人名等）
4. 产出一份"终审报告"：
   - 确认通过的一致性项
   - 仍存在的已知问题
   - 建议的后续优化方向

### 批判性节点的运作原则

1. **制度化，而非可选** — 每个 CP 都必须执行，不能因为"赶时间"或"感觉没问题"就跳过
2. **换视角** — 批判时必须切换到"挑剔评审者"的视角，不能既是作者又是评审
3. **接受被拒绝** — 用户拒绝你的批判意见是正常的，不代表批判没价值
4. **记录所有争论** — 每个批判意见和用户的裁决都要记录，这些争论就是方案的"决策日志"
5. **争论不是对抗** — 批判的目标是让产出更好，不是为了证明谁对谁错

## 关键原则（贯穿全流程）

### 1. 边读边写，不等完美
不要等所有素材到齐再动手。先出草案，审的过程就是深入理解的过程。

### 2. 当专家，不当打字员
主动提批判意见（至少 3-5 条），敢于反对用户的方向。
目标是产出经得起同行评审的内容，不是让用户觉得"AI真听话"。

### 3. 结构先行，内容跟进
先定框架和目录，再填内容。目录确认后再动笔。
避免写到一半发现结构要大改。

### 4. 一致性优先
术语、编号、格式全文档保持一致。
同一个人名在不同地方不能叫不同名字。

### 5. 待确认项显式标记
不确定的地方用 `❓` 标记，不猜测、不编造。
宁可留空，不能写错。

### 6. 文件是记忆
所有产出存文件，不依赖对话上下文。
新 session 能直接从文件接续工作。

### 7. 写入安全
单次 write 不超过 20KB，超过就拆分。内容完整性是目标，大小限制是安全阀。
大文件覆盖前先备份，截断必须立即修复。

### 8. 有争论才有好内容
每个批判性节点（CP1-CP6）都必须执行。
争论不是对抗，是让产出变好的过程。
被用户拒绝批判意见也是正常的——重要的是争论发生了。

### 9. 超出预期是底线
每个核心章节至少包含一个用户没提到但必须考虑的专业洞察。
如果用户审完后说"跟我想的一样"，说明写得不够深。

## 实战经验（从 xmt-sop 项目提炼）

### 框架会多次重构（正常现象）
xmt-sop 项目框架经历了 v1.0→v2.0→v3.0→v4.0→v5.2 共 5 个版本。
每次重构都是认知升级，不要害怕推翻重来。

### 核心章节会越写越厚
01_品种体系从最初的简单描述，最终写到 1299 行。
这是正常的——核心章节承载了最多概念定义和流程描述。

### "六大支撑"这类分组要做二次分层
6 个并列模块太多，人记不住。分成"战略资产层/基础设施层/外部资源层"三层后，
认知负担大幅降低。这是一个通用技巧。

### 驱动引擎和转化出口要用矩阵表达
"内容驱动对 B2C 的影响 ⭐⭐⭐，对 O2O 的影响 ⭐⭐"
用矩阵表达多对多关系，比文字描述清晰 10 倍。

### 口号/编号是团队沟通的锚点
"1+6+2+3"、"选品有道，支撑有力"——这些口号和编号不是装饰，
是整个团队沟通时共用的"地址系统"。投入时间打磨口号是值得的。

### 每卷的开头写清楚"这一卷解决什么问题"
不要直接进内容。先用一句话定位本卷的职责和边界。
后续审稿时可以快速判断"这段内容该不该放在这里"。

## 文件工程规范（贯穿全流程）

> AW 产出的文件集群可能很大（xmt-sop 项目最终约 17000 行），必须建立工程化的文件管理规范，避免截断、丢失、版本混乱。

### 文件写入安全阀

- 单次 `write` 内容不超过 20KB（超出可能截断，内容不完整）
- **20KB 是安全阀，不是质量目标** — 内容该多完整就多完整，不够就拆分
- 预计内容 >20KB → 拆分为多个文件（part1/part2）或用 `exec` + Python 脚本生成
- 预计内容 ≤20KB → 直接写入，无需刻意精简

### 备份规则

覆盖写入已有文件前，满足以下**任一**条件时先备份：
- 文件 >10KB
- 文件包含不可重建的内容（用户提供的素材、会议纪要等）

```bash
cp original.md original.md.bak.$(date +%Y%m%d%H%M%S)
```

### 截断问题处理

**识别信号**：
- AI 响应中出现 "truncated"
- 文件末尾内容不完整（突然中断在段落中间）
- `wc -c` 显示大小远小于预计

**处理步骤**：
1. `tail -20 [file]` 确认截断位置
2. 剩余内容 <20% → 用 `edit` 补充
3. 剩余内容 ≥20% → 重新生成该章节
4. 补完后再次检查文件完整性
5. 记录到 WORKLOG.md

### 版本命名规范

```
v1.0 → v1.1（小改）→ v2.0（重大变更）
```

- 重大修改升大版本号（v4→v5）：结构调整、新增章节
- 小修改升小版本号（v4.0→v4.1）：文字修订、数据更新
- 每次修改记录变更摘要
