# Legacy CSV Migration Plan

References: `data-schema.md` `legacy_csv_projection.v1`.

## Purpose

Move from legacy CSV files to append-only event records without losing auditability or breaking existing read paths.

## Legacy Inputs

- `drafts_{market}.csv`
- `candidates_{market}.csv`
- `four_week_tracker_{market}.csv`
- `target_pool.csv`
- `positions.csv`
- `trade_log.csv`

## v1 Migration Sequence

1. Inventory existing CSV files and compute file hashes.
2. Read each row and compute a stable `legacy_row_hash`.
3. Produce append-only event records with `migration_batch_id`.
   - Normalize generated evidence references to `ev_<ulid>` and claim references to `cl_<ulid>`.
   - If a legacy row contains an evidence-like identifier in another format, preserve it in migration audit fields and emit a new canonical id.
4. Build compatibility projections from events back to CSV shape.
5. Compare projection row counts and key fields against legacy CSV.
6. Keep legacy CSV read-only during compatibility window.
7. Switch SOP readers to event store after S5 dry-run acceptance.

## Rollback

Rollback means restoring readers to legacy CSV and ignoring migrated event batches by `migration_batch_id`. Original CSV rows are not modified during v1 migration.

## Audit Fields

Every migrated event must include:

- `migration_batch_id`
- `legacy_file`
- `legacy_row_number`
- `legacy_row_hash`
- `migrated_at`
- `migration_tool_version`

## Acceptance

- Migration can be rerun idempotently.
- Projection row count matches source row count unless a documented dedup rule applies.
- Free-text reason is preserved as historical note unless tied to evidence.
- No broker or trade action can be generated by migration.
- Migration can resume from a partially completed batch without duplicating events; restart-after-failure must be idempotent.
