# 搜索编排器架构方案（Battle 输入）

## 方案概述

将 multi-search 从线性降级链升级为 Search Orchestrator 架构：
- 一个编排器 + 一个注册器 + 多个独立 provider（每个搜索方案是独立模块）
- 编排器根据搜索场景选择 2-3 个最优 provider，并行/串行发起搜索，聚合去重后返回

## 12 个待决问题

### Q1 编排器运行时
编排器以什么形态存在？Python 脚本？MCP Server？OpenClaw Skill 函数？

### Q2 结果格式统一
不同 MCP Server 返回不同结构的数据。编排器的统一输出格式是什么？

### Q3 搜索需求分类
路由依赖识别场景（中国政策 vs 英文文档 vs 新闻）。谁来做分类？

### Q4 Provider 最小单元
每个搜索方案作为"独立小 skill"，最小单元是什么？MCP 配置片段？SKILL.md？脚本？

### Q5 打分机制
手动评分会过时。打分怎么执行、维护、反映到路由？

### Q6 免费额度追踪
MiniMax 周配额、Brave 月配额。编排器如何知道当前剩余额度？

### Q7 新旧架构关系
线性降级链（现存）和场景路由（新方案）如何共存/替换/合并？

### Q8 Provider 生命周期
MCP Server 进程可能挂掉。编排器如何感知可用性？

### Q9 失败处理策略
3 个 provider 全部失败怎么办？部分成功怎么办？

### Q10 并行 vs 串行
3 个 provider 全部并行/全部串行/混合？谁来决定？

### Q11 成本预算意识
不同 provider 成本不同（免费 vs 配额 vs 付费）。编排器如何做成本优化？

### Q12 注册流程
新增 provider 不改原有代码。注册方式是什么？配置文件？动态扫描？
