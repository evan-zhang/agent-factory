# Ralph Loop v3.1.0 Protocol

本文件是每轮迭代必须读取并遵守的核心循环协议。
只讲规则，不讲安装，不讲配置。
按协议行动，留下可验证结果。

## 1. 文件权限
`PROTOCOL.md`、`PROMPT.md`、`verify.sh` 是只读文件。
不得编辑、重写、格式化、移动、删除这三个文件。
不得通过脚本、批量替换、生成器或间接写入改变这三个文件。
不得修改验证标准来适配你的实现。
不得把计划、理解或中间结论写入 `PROMPT.md`。
需要记录内容时，写入工作目录中的其他文件。
工作目录其他文件可按任务需要修改。
只修改完成本轮目标所需的最小文件集合。
不要改动无关文件，不要做未请求的重构、格式化或依赖变更。
发现只读文件确实需要变更时，停止本轮实现，记录原因。

## 2. 每轮入口
先读取 `state.json`，确认 mode、phase、route、checklist、journal 和上轮结果。
再读取 `PROMPT.md`，确认任务目标、范围、验证方式和可选阈值。
最后读取本文件。
冲突时，按本文件、`PROMPT.md`、`state.json` 的顺序判断优先级。
`state.json` 损坏或缺关键字段时，不要猜测；先记录问题，再做最小修复或停止本轮。
每轮只选择一个主要任务。
任务过大时，缩小到可在本轮验证的最小切片。

## 3. 取任务
执行者模式下，从 checklist 中选择第一个未完成项。
自主者模式下，从 route 中选择下一步未完成项。
route 不足以指导行动时，先补充工作目录中的计划文件。
不要修改 `PROMPT.md` 来补充计划。
不要一次推进多个无关目标。
本轮开始前，写清楚本轮意图。
本轮意图必须能被 `verify.sh` 或明确证据验证。
若所有项都已完成，运行 `verify.sh` 做最终确认；通过后将 state 标记为 done。

## 4. 执行
先检查当前 git 状态，识别已有未提交改动。
不要覆盖不属于本轮的已有改动。
编辑前先理解相关文件。
优先删除、简化、复用现有逻辑。
其次修改局部实现。
最后才新增结构、依赖或复杂抽象。
保持 diff 小而集中。
不要为未来可能需求写代码。
不要创建无法由本轮验证覆盖的代码。
不要把临时调试输出留在最终结果中。
必要时更新工作目录中的说明文件或 state 进度。

## 5. 验证
每轮改动后必须运行 `bash verify.sh`。
只相信 `verify.sh` 的结果。
额外测试、lint 或检查命令可以运行，但不能替代 `verify.sh`。
`verify.sh` 最后一行必须是 JSON。
JSON 必须形如：`{"status":"PASS","score":0.234,"details":"..."}`。
`status` 表示 PASS 或 FAIL。
`score` 越小越好。
`details` 说明关键证据或失败原因。
最后一行不是合法 JSON 时，本轮视为验证失败。
退出码 0 表示整体验证通过；退出码 1 表示整体验证失败。
没有连续分数的任务使用离散分数：PASS 的 score 为 0，FAIL 的 score 为 1。
不要发明第三种退出语义。

## 6. 比较
记录改动前 score。
记录改动后 score。
比较 score_before 和 score_after。
更小的 score 是更好的结果。
PASS 优先于 FAIL。
同为 PASS 时，选择更简单的实现。
同为 FAIL 时，只保留能推进下一轮的诊断记录。
无法证明有进展时，撤回本轮改动。
不要保留让验证更差的改动。
不要保留只改善表象但不改善目标的改动。

## 7. Simplicity Criterion
删代码、删配置、删依赖并得到同样或更好结果，是最佳迭代。
同样效果下，选择文件更少、路径更短、状态更少的方案。
同样效果下，选择更容易被 `verify.sh` 覆盖的方案。
新增代码必须用明确收益证明。
新增代码但只有微小改善时，应撤回。
微小改善的阈值由 `PROMPT.md` 配置。
本协议不硬编码阈值。
`PROMPT.md` 未配置阈值时，按 PASS/FAIL 和可解释进展判断。
不要用复杂度换取不可解释的小幅 score 变化。
若简单方案和复杂方案都通过，保留简单方案。

## 8. 保留、撤回、提交
验证通过且 score 改善，保留本轮改动。
验证通过且 score 不变但实现更简单，保留本轮改动。
验证通过但实现更复杂，只有在收益超过阈值时保留。
验证失败且没有明确诊断价值，撤回本轮改动。
验证失败但产生有用诊断，只保留诊断记录。
只读文件被修改时，本轮必须视为失败，恢复只读文件并撤回本轮改动。
保留改动前，查看 diff，确认只包含本轮意图。
确认没有只读文件变更。
确认没有无关格式化、临时文件、日志或缓存。
保留改动时，提交本轮改动，提交信息写明本轮目标和验证结果。
撤回改动时，回到本轮开始前状态。
不得撤回用户或其他进程已有改动。
无法区分已有改动和本轮改动时，停止并记录风险。

## 9. results.tsv
每轮必须追加一行结果到 `results.tsv`。
第一行 schema 为：`iteration	score_before	score_after	status	diff_stat	reason`。
字段使用 tab 分隔；不要使用逗号分隔。
不要写多行 reason。
`iteration` 是本轮编号。
`score_before` 是改动前分数。
`score_after` 是改动后分数。
`status` 只能是 keep、discard 或 crash。
`diff_stat` 是本轮 diff 的简短统计。
`reason` 是一句话，说明保留、撤回或崩溃原因。
保留成功改动时，status 写 keep。
撤回本轮改动时，status 写 discard。
只读文件被篡改、验证器异常或执行中断时，status 写 crash。
results 记录必须反映真实结果。
不要为了显得进展顺利修改历史记录。

## 10. state.json
state.json 的完整字段规范见 SKILL.md，本节只规定更新规则。
每轮结束前更新 state.json。
只记录真实完成的事项。
不要提前把未验证事项标记完成。
验证通过后，才能标记 checklist 项完成。
自主者模式下，journal 记录本轮意图、动作、验证结果和下一步。
执行者模式下，记录完成项和证据。
若撤回本轮代码，仍可记录学到的失败原因。
state 更新必须和 results 记录一致。

## 11. 反模式
同一文件连续修改 3 次以上仍无改善，停止继续微调。
换方案前，重新阅读目标、相关代码和验证输出。
验证反复失败时，不要继续猜；重新读取 `PROMPT.md`、`state.json`、失败日志和最小相关源码。
无进展时，缩小范围，把目标缩到一个函数、一个文件或一个测试。
diff 变大但 score 不改善时，撤回。
新增抽象但无法删除旧复杂度时，撤回。
为了通过验证而削弱需求时，撤回。
修改测试或验证器来通过任务时，撤回。
绕过 `verify.sh` 时，撤回。
无法解释保留理由时，撤回。

## 12. 停止条件
达到本轮目标并通过验证后停止本轮。
所有 checklist 或 route 项完成后，运行最终验证。
最终验证 PASS 后，将 phase 标记为 done。
遇到权限冲突、只读文件变更或状态损坏时停止本轮。
遇到连续无进展时停止扩大改动，改为记录诊断。
停止时必须留下三类证据：diff、verify 结果、results.tsv 记录。
下一轮从 `state.json` 和 `results.tsv` 继续。
