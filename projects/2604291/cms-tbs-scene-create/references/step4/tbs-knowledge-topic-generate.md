### 产品知识主题生成（Step4.1，Step3 完成后执行）

**步骤**：在 Step3 的“标题 + 场景背景”已生成并经用户确认之后执行。由 Agent 基于 Step3 已确认的 `医生核心顾虑` 与 `代表目标`，推荐 2-4 条“产品知识主题”短语，并回显给用户逐条确认/修改。

本文件只定义“输入/输出字段”和“生成/禁止规则”；实际主题正文与是否存在性匹配由后续 Step4 的确认与后台校验流程承担。

#### 执行时机（按需加载）
- 必须在 `references/step3/tbs-scene-validate.md` 中定义的确认动作全部完成后执行（含：完整已确认清单、`title`、`sceneBackground` 以及变更影响字段的二次确认）。
- 仅在需要“生成/刷新产品知识主题”时执行。
- 本步骤结束后进入“用户逐条确认主题 -> 主题存在性匹配 -> 决策是否允许落库”。

#### 4.4.1 输入
- `businessDomainName`：业务领域名称（已确认）
- `departmentName`：科室名称（已确认）
- `drugName`：品种/产品名称（已确认）
- `location`：场景时间地点（已确认）
- `doctorConcerns`：医生核心顾虑（已确认，建议 1-2 条）
- `repGoal`：代表目标（已确认）

#### 4.4.2 输出
- `scene.productKnowledgeNeeds`：产品知识主题短语数组
  - 数量：2-4 条
  - 每条：短语字符串，可直接展示给用户删改确认
- `scene.knowledgeIds`：按匹配规则命中的知识 ID 数组

#### 4.4.2.0 用户回显确认（必须）
在让用户确认本次产品知识主题时，输出顺序固定为：
1. 先回显“此前已确认的完整信息清单”（不能丢字段）：
   - 业务领域、科室、产品、时间地点
   - 医生核心顾虑、代表目标
   - 医生画像、最佳实践要点
   - 已确认的标题、场景背景
2. 再回显“本次待确认的 `scene.productKnowledgeNeeds`”。
3. 用户修改主题后，重复以上回显顺序再次确认，直到用户明确确认无误。

#### 4.4.2.1 主题匹配上下文知识（必须）
基于 `scene.productKnowledgeNeeds` 到 Step1 已写入的 `state.productKnowledges` 中查找：
- **第一轮：精确匹配**
  - 匹配条件：`drugId + title`
    - `drugId`：取当前已确认产品对应的 `drugId`
    - `title`：取每条产品知识主题文本（trim 后精确匹配）
  - 命中：记录对应 `knowledgeId`，写入 `scene.knowledgeIds`
- **第二轮：模糊匹配降级**（仅对精确匹配未命中的主题执行）
  - 匹配条件：`drugId` 一致 + `title` 包含匹配（双向）
    - 用户主题文本包含知识库 title，或知识库 title 包含用户主题文本
    - 匹配时忽略空格和标点差异
  - 命中：记录对应 `knowledgeId`，写入 `scene.knowledgeIds`
- **仍未命中**：
  - 不写入该条 `knowledgeId`
  - 明确提示用户该主题未找到（并引导用户修改主题或确认缺失处理）

#### 4.4.3 生成原则（主题要覆盖知识缺口）
1. 医生核心顾虑：围绕医生为什么不接受/担心什么来选主题。
2. 代表目标：围绕这次沟通希望推动的行为变化来选主题。
3. 产品价值：围绕产品临床价值、定位、适用患者或证据方向来选主题。
4. 场景执行：围绕院外可及性、患者衔接、流程、合规边界中最相关内容来选主题。

#### 4.4.4 禁止
1. 不得直接复述医生顾虑原文作为主题。
2. 不得输出“产品核心信息”“常见问题”等空泛占位词。
3. 不得编造疗效结论、指南结论或证据出处。
4. 不得把知识正文/话术/证据段落写入“主题”里（主题只做短语）。
5. 不得输出“当前未提供正文”“暂无正文”等状态提示。
6. 不要把后续“scenario-json-parse 阶段”的新主题字段/结构当作主题输出。

#### 4.4.5 输出示例（输入->推荐主题）
输入要点：医生担心处方外流安全性；代表目标是增加院外处方数量；场景强调临床价值与院外可及性。  
建议主题：
- 产品临床价值与适用患者
- 处方外流安全性与风险沟通
- 院外购药可及性与患者用药衔接
- 医生院外处方顾虑回应要点
