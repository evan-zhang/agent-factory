## Skill 需求文档（用于编写“医药代表 TBS 训战场景”）

### 1. 背景与目标
为医药代表提供可训练、可复用的 `TBS 训战场景` 编写能力。Skill 需要在 Step1-4 的链路中，引导用户补齐必要信息，随后自动生成并校验：
- 场景基础内容（时间地点、医生画像、担忧点、代表目标、最佳实践）
- 场景呈现结构（标题、角色画像设定、教练专属设定）
- 产品知识主题（供用户确认；若缺失则触发工作汇报发送流程）
- 最终提交并落库（当所有条件满足时）

> 原则：**简洁至上**。Skill 应优先用少量字段完成闭环，而不是冗长对话。

### 2. Skill 输入（上下文/用户提供）
Skill 需要从上下文或用户输入获取以下信息（按需分步补齐）：

- Step1：拉取并锁定训练上下文（领域/科室/品种）
  - `业务领域`
  - `科室`
  - `品种`
- Step2：交互回显确认模板 + 补问收集（用户互动）
  - `用户实际交互内容`（按模板引导用户填写）
- Step3：校验并结构化场景要素（医生画像/担忧/目标/最佳实践）
  - `场景发生的时间地点`
  - `医生担忧点（关注/顾虑）`
  - `代表目标（希望达成）`
  - `医生画像内容`：人设配置、简介描述、姓氏、职称
  - `最佳实践要点`（由用户提供并完成确认：开场话术/回应问题话术/推荐建议）
- Step4：生成场景交付稿 + 产品知识确认（缺失则上报）
  - 基于 `业务领域/科室/品种 + 场景基础内容` 生成场景内容
  - 基于 `业务领域/科室/品种 + 场景基础内容` 生成 `产品知识主题清单`
  - `产品知识确认结果`（已确认/缺失原因）
  - 若缺失：触发“发送工作汇报给对应人员”的动作

### 3. Skill 输出（落库前的交付物）
Skill 至少应生成/维护以下输出结构（字段名需与落库 schema 对齐）：

- `scenarioBase`（场景基础内容）
  - 业务领域
  - 科室
  - 品种
  - 时间地点
  - 医生担忧点（关注/顾虑）
  - 代表目标
  - 医生画像（含人设配置、简介描述、姓氏、职称）
  - 最佳实践要点（由用户提供并完成确认）
  
- `scenarioPresentation`（场景呈现内容）
  - 场景标题
  - 角色画像设定内容
  - 教练专属设定内容
    - 最佳实践要点（用于教练话术/提示引用）
- `productKnowledgeTopics`
  - 产品知识主题清单（逐条：已确认/缺失）
  - 缺失原因（若缺失）
- `submission`
  - 提交条件是否满足
  - 提交结果/落库状态
  - （缺失知识场景）缺失上报载荷：科室、品种、缺失的主题、缺失原因、场景上下文摘要、期望补充信息

### 4. 核心流程（Step1-4 + 校验与提交）
每一步尽量减少多轮问答，只向用户补齐“缺失项”。

#### Step 1：拉取并锁定训练上下文（领域/科室/品种）
要做的事：
1. 从用户选择或固定配置中获取 `业务领域/科室/品种`
2. 将它们放入后续对话上下文，作为生成与知识主题的主键条件

产出：
- 上下文锁定字段：`业务领域 + 科室 + 品种`

#### Step 2：交互回显确认模板 + 补问收集
要做的事：
1. 当用户表达创建意图且信息不完整时：直接展示默认输入模板原文（来自 `cms-tbs-scene-create/references/step2/default-input-template.md`），并说明“信息不完整也没关系，我会回显确认清单并补问缺口”。
2. 从用户回复中提取“确认清单”（业务领域/科室/品种、时间地点、医生担忧点、代表目标、医生画像四要素、最佳实践要点三项、以及 `用户实际交互内容`），并回显给用户确认。
3. 仅补问 Step3 仍缺的字段；当是“Step3 校验失败后返回 Step2”时，不重新展示默认输入模板，只输出缺失项与补问问题。

产出：
- `用户实际交互内容`

#### Step 3：校验并结构化场景要素（医生画像/担忧/目标/最佳实践）
要做的事：
1. 校验用户提交内容是否满足最低条件（不满足则退回补全）
2. 画像认识必须由用户提供并完成确认（人设配置、简介描述、姓氏、职称）
3. 自动整理并要求补齐/确认以下内容：
   - 地点
   - 医生担忧点（关注/顾虑）
   - 代表目标（希望达成）
   - 医生画像：人设配置、简介描述、姓氏、职称（用户提供并确认）
   - 最佳实践要点（由用户提供并完成确认：开场话术/回应问题话术/推荐建议）
4. 若 Step3 任意校验项不满足：回到 Step2 继续补全（仅回补缺失字段：时间地点、医生担忧点、代表目标、医生画像、最佳实践要点）

产出：
- `scenarioBase`
- 医生画像结构化字段

#### Step 4：生成场景交付稿 + 产品知识确认（缺失则上报）
要做的事：
1. 按“生成场景内容”的结构生成：
   - 场景标题
   - 角色画像设定内容
   - 教练专属设定内容
2. 基于 `scenarioBase + 业务领域/科室/品种` 生成 `产品知识主题清单`（作为待确认/可编辑的初稿）
3. 产品知识确认由用户逐条进行：用户可以**修改主题内容**；用户最终确认后的主题，系统再按 `科室 + 品种 + 产品知识主题` 检查是否存在：
   - 若存在：进入提交落库流程
   - 若存在与否仍存在缺失：触发“发送工作汇报给对应人员”流程，并标记为不允许落库
     - 对应人员：按 `科室 + 品种` 查找
     - 上报载荷：科室、品种、缺失的主题、缺失原因、场景上下文摘要、期望补充信息
4. 当最终确认的产品知识主题全部匹配存在时：执行“提交去落库”

产出：
- `scenarioPresentation`
- `productKnowledgeTopics`（含确认结果/缺失处理）
- `submission`（含缺失上报状态/不允许落库原因）

### 5. 校验规则（简洁版）
提交是否允许落库应满足：
- 业务领域已明确
- 科室已明确
- 品种已明确
- 时间地点已明确
- 医生担忧点（关注/顾虑）已明确
- 代表目标已明确
- 医生画像四要素齐全：人设配置、简介描述、姓氏、职称
- 最佳实践要点已由用户提供并确认
- 产品知识清单已完成确认，或缺失项已走“工作汇报”流程且不允许落库

### 6. 交互风格与输出约束
- 每步只问“缺失项”，其余不反复确认
- 每次输出只展示当前步骤需要的内容块
- 对于产品知识主题：先列主题，再让用户确认；用户可修改主题后需再次确认，若仍缺失则走汇报且不允许落库

### 7. 交付物清单（对 Skill 的验收点）
- 能完成 Step1-4 的完整生成链路
- 能在 Step3 做到严格校验并引导补齐
- 能生成 Step4 的结构化场景呈现内容
- 能生成产品知识主题并让用户确认；缺失时触发汇报
- 能在所有条件满足时触发提交落库动作
